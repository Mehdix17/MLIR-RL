\chapter*{General Conclusion}
\addcontentsline{toc}{chapter}{General Conclusion}
\label{ch:conclusion}

This thesis proposed and evaluated three architectural enhancements to a reinforcement-learning-based auto-scheduler for the \gls{mlir} compiler framework. The starting point was the prior \gls{mlir}-\gls{rl} agent by Tirichine et al.~\cite{tirichine2025reinforcement}, which used an \gls{lstm}-based policy and a sparse terminal speedup reward. We identified and systematically addressed three limitations in this baseline: hardware-agnostic observations, sequential \gls{lstm} state encoding, and the reliance on unverified intermediate shaped rewards. To achieve hardware awareness, we introduced a 7-element hardware feature vector that conditions tile sizing and parallelization choices directly on the execution platform. For program state representation, we replaced the sequential encoder with a Transformer model to capture long-range structural dependencies across non-adjacent loop coordinates via self-attention layers. Finally, we investigated whether dense intermediate reward signals based on static analysis could accelerate policy convergence. The empirical results demonstrated that static structural heuristics are unreliable proxies for physical runtime performance, as the agent learned to exploit the heuristic boundaries rather than improve actual execution speed. Removing these shaped components entirely and relying solely on the terminal execution speedup yielded our best-performing agent configuration, which successfully scaled optimization support to a wide variety of complete neural network architectures.

Despite these advancements, five core limitations define the current scope of this framework. First, the environment loop remains bottlenecked by the overhead of just-in-time compiling and physically executing every candidate transformation sequence, which limits the speed of exploration during training. Second, the total removal of shaped rewards leaves the agent entirely dependent on these physical runtime measurements. Developing a learned cost model to approximate execution behavior represents a critical missing component to bridge this gap. Third, the empirical evaluation was restricted to \gls{cpu} microarchitectures; modern \gls{gpu} targets introduce completely distinct execution constraints—such as thread concurrency bounds and memory coalescing matrix topologies—that are not yet modeled. Fourth, the transformation space is restricted to five fundamental loop operation types, omitting optimizations like loop unrolling, padding, and data packing. Fifth, the inference framework relies on a greedy action selection strategy that lacks the ability to backtrack or dynamically evaluate alternative optimization trajectories.

Several future research lines follow directly from these constraints. Integrating a learned cost model to predict program speedup directly from structural state characteristics could replace physical execution cycles during training, decoupling policy search from compiler backend latency. Extending the environment to model execution constraints on accelerator platforms via \gls{mlir}'s native \gls{gpu} dialects would rigorously test the cross-architecture validity of the system. Expanding the available action space to include unrolling, packing, and memory alignment transformations would unlock performance gains on highly irregular operations. Finally, replacing greedy execution with tree-search or beam-search inference mechanisms would enable the policy to discover complex, non-obvious optimization pipelines by exploring parallel paths.

In summary, this work advances the compiler optimization trajectory established by Bendib et al.~\cite{bendib2024reinforcement} and Tirichine et al.~\cite{tirichine2025reinforcement}, moving learned loop scheduling from single isolated operations toward comprehensive state encoding and architectural adaptability. The structural insights gained from this study underscore three design principles for future compiler systems: sparse terminal speedups provide a cleaner optimization target than static proxy metrics, self-attention mechanisms capture deep loop nesting relationships more effectively than sequential models, and explicit hardware conditioning requires localized exposure or fine-tuning to achieve effective zero-shot cross-platform generalization. Resolving the remaining execution, action space, and inference constraints defines the immediate path toward fully autonomous, architecture-adaptive compiling infrastructures.
