\section{Introduction}

The previous chapter described the design of our proposed \gls{rl} auto-scheduler. This chapter empirically evaluates the system to measure the impact of its three architectural enhancements: hardware-aware observations, shaped rewards, and the transformer-based encoder. We first conduct a controlled ablation study that trains four agent variants, each disabling a single component to isolate the contribution of each enhancement. The best-performing variant is then selected for head-to-head comparisons against two baselines: the standard \gls{mlir} heuristics (\gls{mlir}-Base) and the prior \gls{mlir}-\gls{rl} agent~\cite{tirichine2025reinforcement}.

The software stack consists of the following:

\subsection{Evaluation Metrics}

Each optimized benchmark is executed once on the target hardware. The primary evaluation metric is the execution-time speedup, defined as:

\begin{equation}
    \text{Speedup} = \frac{T_\text{\gls{mlir}-Base}}{T_\text{optimized}}
\end{equation}

$T_\text{\gls{mlir}-Base}$ is the execution time of the standard \gls{mlir} compilation pipeline but without any \gls{rl}-guided transformations. A speedup above $1.0$ indicates improvement over the standard compiler.

For aggregated comparisons across multiple benchmarks, we report the geometric mean of per-benchmark speedups, the standard metric for normalized ratios that resists skew from outliers:

\begin{equation}
    \text{Geometric Mean Speedup} = \left( \prod_{i=1}^{n} \text{Speedup}_i \right)^{1/n}
\end{equation}

\subsection{Datasets}

We use two evaluation sets : \\ \\
\textbf{Single-operation set:} contains 819 benchmarks covering four operation types: matrix multiplication (232 benchmarks), 2D convolution (71 benchmarks), element-wise ReLU operations (106 benchmarks), and pooling (315 benchmarks). These benchmarks are extracted from 18 neural network architectures spanning vision, language, and graph models.

\textbf{Full-model set:} contains 918 operation blocks extracted from the same 18 architectures. Blocks are windows of 5 consecutive operations extracted with a stride of 3. Each model contributes between 11 and 150 blocks. All blocks are drawn from the training set, as blocks were placed entirely in training to prevent data leakage through overlapping extraction windows (Section~4.2).

\section{Ablation Study}
\label{sec:ablation-multi-hw}

To isolate the contribution of each architectural enhancement, we trained four agent variants under identical \gls{ppo} hyperparameters, differing by exactly one component:

\begin{enumerate}
    \item \textbf{Full Agent:} all three novelties enabled: hardware-aware observation, shaped reward (arithmetic intensity, vectorizability, parallel-loop ratio), and Transformer encoder.
    \item \textbf{No-Reward-Shaping Agent:} hardware-aware observation and Transformer encoder, relying solely on the terminal speedup signal without intermediate shaped rewards.
    \item \textbf{No-Transformer Agent:} hardware-aware observation and shaped reward, but using an \gls{lstm} encoder instead of the Transformer.
    \item \textbf{No-Hardware-Features Agent:} shaped reward and Transformer encoder, but with all hardware features set to zero (hardware-agnostic).
\end{enumerate}

All 4 variants were trained on the Bergamo cluster for 10{,}000 iterations and evaluated on the full-model set. \\ \\
Figure~\ref{fig:ablation-study} presents per-model speedups across 18 model families for all four variants.

\begin{figure}[H]
    \centering
    \includegraphics[width=1.0\textwidth]{ressources/chapter5/ablation_study.png}
    \caption{Per-model ablation study comparing all four agent variants across 18 models}
    \label{fig:ablation-study}
\end{figure}

\textbf{Analysis:} Figure~\ref{fig:ablation-study} reveals a clear ranking. The No-Reward-Shaping agent achieves the highest overall speedups, outperforming the Full Agent on the majority of models. On Albert~\cite{lan2020albert}, the No-Reward-Shaping agent reaches 7.99x while the Full Agent trails at 3.85x. On BART, the gap is 6.72x versus 3.33x. This result indicates that the static heuristics used for shaped rewards---arithmetic intensity, vectorizability, and parallel-loop ratio---are imperfect proxies for actual execution speedup. The agent learns to maximize these heuristics rather than the true objective, producing suboptimal schedules. Removing shaped reward entirely and relying solely on the terminal speedup signal produces a more robust policy.

The No-Transformer agent (\gls{lstm} encoder) underperforms the Full Agent on most models, confirming that the Transformer encoder provides a consistent advantage. The gap is most visible on deep loop nests such as those in T5~\cite{raffel2020t5} and Llama~3.2~1B~\cite{llamateam2024llama3}, where long-range dependencies between outer tiling loops and inner vectorization targets matter. For shallow nests common in simpler operations, the two encoders perform comparably. The Transformer's self-attention mechanism processes all loop levels simultaneously, preserving hierarchical relationships that the \gls{lstm}'s sequential encoding can degrade.

The No-Hardware-Features agent trails on most models, particularly on Albert~\cite{lan2020albert} and BART~\cite{lewis2020bart}, where cache-sensitive tiling decisions are critical. Without hardware context, the agent cannot condition its tile size choices on the available L1/L2/L3 cache sizes, leading to suboptimal cache utilization.

The No-Reward-Shaping agent emerges as the best-performing variant and is selected for all subsequent experiments (Sections~5.4 and~5.5). We refer to this variant as \textbf{Our Agent} in the remainder of this chapter.

\subsection{Hardware-Aware Observation: Multi-Platform Analysis}

To test whether hardware features enable cross-platform generalization, we evaluate the impact of hardware-awareness by comparing two configurations of our agent: one receiving the full 7-element hardware vector (cache sizes, core counts, \gls{simd} width, clock speed), and one receiving a zero-filled hardware vector. We evaluate on three clusters : Bergamo is the training cluster (AMD EPYC 9754, 256 cores). Dalma uses a fundamentally different architecture (Intel Xeon E5-2680 v4, 28 cores). Jubail shares the AMD EPYC lineage but has half the cores (EPYC 7742, 128 cores). This cross-platform setup tests whether hardware features enable zero-shot transfer.

\end{figure}

\textbf{Analysis:} The hardware-aware agent consistently achieves the highest geometric mean speedups across the majority of the 18 evaluated architectures. The performance gains are significantly pronounced in transformer-based language models, such as Albert~\cite{lan2020albert}, T5~\cite{raffel2020t5}, and Llama~\cite{llamateam2024llama3}, which are dominated by repetitive patterns of multi-head attention blocks and heavy batch matrix multiplications. In these scenarios, the presence of explicit hardware feature -observations enables the \gls{rl} policy to accurately select optimal multi-level tiling factors. This prevents cache thrashing on the 256-core architecture. For traditional convolutional vision networks like ResNet~\cite{he2016resnet} and VGG-16~\cite{simonyan2015vgg}, the agent similarly achieves a robust speedup profile, demonstrating that the learned scheduling policy successfully targets complex loop dimensions without destabilizing the downstream lowering pipeline of the MLIR compiler.

\textbf{Analysis:} The results demonstrate varying performance deltas between the hardware-aware agent and the configuration without hardware features depending on the specific operator type. For \texttt{Pooling} operations, the hardware-aware configuration obtains a 3.00x speedup compared to 1.18x for the baseline configuration. This variation indicates that explicit hardware parameters provide information that influences the selection of tiling sizes and vectorization factors in multi-dimensional nested loops. Conversely, the performance divergence decreases for element-wise and lower-dimensional operators. For \texttt{ReLU}, the configurations achieve speedups of 2.1x and 1.3x, respectively. For \texttt{Matmul}, the performance profiles are comparable, with the configuration without hardware features reaching 2.1x and the hardware-aware agent reaching 2.0x. This convergence suggests that for certain structural dimensions, the optimization schedules generated by the policy network remain effective even in the absence of explicit hardware features.

However, the performance delta between the configuration with hardware features and the configuration without hardware features is narrower on the Jubail cluster than the gaps observed on the Bergamo cluster. When specific loop transformations are hard-conditioned on native microarchitectural traits, they lose their mathematical optimality on different execution pipelines. This confirms that while the high-level, hardware-invariant scheduling choices captured by the Transformer encoder transfer effectively across different processor vendors, the explicit inclusion of hardware parameters introduces a strict dependency that can harm optimization efficiency outside the native training environment. \\

In summary, the ablation study establishes three conclusions. First, the No-Reward-Shaping agent---relying solely on the terminal speedup signal---outperforms all shaped-reward variants, and is selected as the best configuration. Second, the Transformer encoder provides a consistent advantage over the \gls{lstm} encoder, especially on deep loop nests. Third, hardware-aware features deliver their largest benefit on the training platform and degrade gracefully on dissimilar hardware, never hurting performance.

Both agents were trained on the Bergamo cluster for 10{,}000 iterations and evaluated on a test set of unseen benchmarks. Our agent uses hardware-aware observation and the Transformer encoder, relying solely on the terminal speedup signal. The prior agent uses none of these enhancements.

\textbf{Analysis:} Figure~\ref{fig:overall-performance} shows the geometric mean speedup per operation type for both agents. The hardware-aware features further enable the agent to select tile sizes and parallelization strategies appropriate for the Bergamo hardware.

\section{Summarizing Main Findings}
\label{sec:main-findings}
The empirical evaluation of the auto-scheduling framework highlights four primary insights regarding reinforcement learning within multi-hardware compiler environments:

First, \textbf{terminal speedup alone is sufficient for learning.} The agent trained without reward shaping consistently outperformed all shaped-reward configurations across the evaluated model families. Intermediate code-analysis heuristics tend to mislead the policy because they reward structural properties that are necessary but not sufficient for execution throughput; a loop nest can display optimal arithmetic intensity on paper while compiling to highly inefficient machine code.

Second, \textbf{hardware-aware features reveal the significant difficulties of zero-shot generalization.} Testing the framework on an entirely unfamiliar processor architecture demonstrates that robust zero-shot generalization remains exceptionally hard to achieve in machine learning compiler studies. Without explicit retraining, the hardware-conditioned variant over-constrains the policy when executing transformations on an unseen hardware configuration, occasionally causing it to underperform compared to a hardware-agnostic baseline. However, despite these zero-shot transfer limitations, utilizing explicit hardware characteristics remains highly promising; exposing the agent to diverse target parameters through localized training or fine-tuning across different hardware architectures provides a clear path for the policy to learn adaptive, hardware-specific optimization invariants.

Third, \textbf{broad support for end-to-end neural network model optimization is feasible at scale.} While the prior sequential agent was structurally constrained and limited to a very few set of supported networks, our proposed framework successfully extends compilation support to a much larger and more varied set of neural network models featuring completely different layers and macro-architectures. This demonstrates that learned optimization strategies can scale seamlessly from isolated kernels to complete, complex deep learning inference pipelines. Our design intentionally prioritizes structural coverage and compiler stability across these diverse multi-layered configurations, trading away peak performance on single regularized algebra loops to ensure that complete modern topologies compile successfully without triggering illegal transformation chains.

Fourth, \textbf{the Transformer encoder provides systematic advantages over sequential models on deep loop structures.} The self-attention layer successfully preserves structural information across non-adjacent loop levels. This capability is critical for large language models such as T5 and Llama~3.2~1B, where deep loop hierarchies contain the primary computational bottlenecks. For shallow loop nests common in basic element-wise operations, the Transformer and sequential \gls{lstm} encoders produce comparable optimization results.

\section{Conclusion}
\label{sec:conclusion}
This chapter evaluated the proposed reinforcement learning auto-scheduler through systematic ablation studies and baseline comparisons across multiple hardware clusters. The experimental results validate that combining a Transformer-based loop-nest encoder with hardware-aware observations while relying exclusively on raw terminal speedup rewards, yields the most robust and performant configuration. This optimized agent successfully scales from isolated operator kernels to complete neural network pipelines, delivering significant execution speedups on complex, transformer-based architectures while maintaining strict structural safety constraints during compilation.
