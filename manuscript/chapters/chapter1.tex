\chapter{Compiler and Code Optimization}
\label{ch:compiler-optimization}

\section{Introduction}

A compiler is a program that translates source code written in a high-level programming language into machine code executable by a processor. This translation is not a straightforward one-to-one mapping: modern compilers perform extensive analysis and transformation of the program to produce code that executes faster, consumes less energy, or occupies less memory. These transformations are collectively referred to as \textit{code optimizations}.

The importance of compiler optimization has grown alongside the complexity of both hardware and software. Modern processors feature deep cache hierarchies, wide \gls{simd} units, and multiple cores, each introducing new opportunities and constraints for program optimization. Simultaneously, software workloads, particularly in machine learning, have grown in scale and computational intensity, making the quality of compiled code a critical factor in overall system performance.

\section{Phases of a Compiler: From Source Code to Executable}

A compiler typically operates in three major phases: the \textit{front-end}, the \textit{middle-end}, and the \textit{back-end}~\cite{aho2006compilers}. Each phase transforms the program into a progressively lower-level representation.

    \caption{The three phases of a compiler.}
    \label{fig:compiler-pipeline}
\end{figure}

\subsection{Front-End}

The front-end is responsible for reading the source code and converting it into an intermediate representation (\gls{ir}). This process involves three sub-steps:

\begin{itemize}
    \item \textbf{Lexical analysis} (scanning): The source text is broken into tokens, keywords, identifiers, literals, and operators. For instance, the expression \texttt{x = a + 2} is tokenized into \texttt{IDENT(x)}, \texttt{ASSIGN}, \texttt{IDENT(a)}, \texttt{PLUS}, \texttt{INT(2)}.
    \item \textbf{Syntax analysis} (parsing): The sequence of tokens is organized into a parse tree or abstract syntax tree (\gls{ast}) according to the grammar of the source language. This step verifies that the program is syntactically well-formed.
    \item \textbf{Semantic analysis}: The compiler checks for semantic correctness, type compatibility, variable declarations before use, scope rules, and annotates the \gls{ast} with type information and symbol table entries.
\end{itemize}

The output of the front-end is an \gls{ir}, typically in a form such as three-address code or static single assignment (\gls{ssa}) form. \gls{ssa} is a representation where each variable is assigned exactly once, which simplifies many subsequent analyses and transformations.

\subsection{Middle-End}

The middle-end operates on the \gls{ir} and is where the majority of optimizations take place. Because the \gls{ir} is independent of both the source language and the target machine, optimizations at this level are portable and can be shared across different language front-ends and hardware back-ends. Middle-end optimizations include:

\begin{itemize}
    \item \textbf{Control-flow optimizations}: constant propagation, dead code elimination, loop-invariant code motion.
    \item \textbf{Data-flow optimizations}: common subexpression elimination, copy propagation, strength reduction.
    \item \textbf{Loop optimizations}: tiling, fusion, interchange, unrolling, vectorization (detailed in Section~1.4).
    \item \textbf{Inter-procedural optimizations}: function inlining, inter-procedural constant propagation.
\end{itemize}

The transformations in our reinforcement learning agent operate at this middle-end level, specifically on the \gls{mlir} Linalg dialect, which represents structured linear algebra operations on buffers and tensors.

\subsection{Back-End}

The back-end translates the optimized \gls{ir} into machine code for a specific target architecture. This phase includes:

\begin{itemize}
    \item \textbf{Instruction selection}: mapping \gls{ir} operations to target machine instructions, often using pattern-matching techniques on a tree representation of the \gls{ir}.
    \item \textbf{Instruction scheduling}: reordering instructions to exploit instruction-level parallelism and avoid pipeline stalls.
    \item \textbf{Register allocation}: assigning program variables to the limited set of physical registers, spilling to memory when necessary.
    \item \textbf{Code emission}: generating the final assembly or object code.
\end{itemize}

While the back-end performs critical low-level optimizations, our work focuses on the middle-end, where high-level loop transformations have the largest impact on memory access patterns and parallelism.

\section{Optimization Paradigms and Techniques}

Compiler optimizations can be categorized along several axes, and they are supported by fundamental program analysis techniques.

\subsection{Machine-Dependent vs. Machine-Independent optimization}

Optimizations are classified as \textit{machine-independent} when they improve the program without knowledge of the target hardware, and \textit{machine-dependent} when they exploit specific hardware features.

Machine-independent optimizations include constant folding (evaluating constant expressions at compile time), dead code elimination (removing computations whose results are never used), and common subexpression elimination (reusing previously computed values). These transformations are universally beneficial and apply regardless of the target architecture.

Machine-dependent optimizations, by contrast, require knowledge of the target processor. Examples include selecting vector instructions matching the \gls{simd} width of the \gls{cpu}, choosing tile sizes that fit within a specific cache level, and scheduling instructions to match the pipeline depth of a particular micro-architecture. One of the contributions of this thesis (version V1, described in Chapter~4) is to make the \gls{rl} agent hardware-aware by feeding processor specifications directly into its observation.

\subsection{Static vs.\ Dynamic Optimization}

Optimizations can also be classified by when they are applied. \textit{Static optimization} occurs at compile time, before the program executes. The compiler analyzes the source code or \gls{ir} and applies transformations based on information available statically. Most traditional compiler optimizations fall into this category.

\textit{Dynamic optimization} occurs at runtime, during program execution. Just-In-Time (\gls{jit}) compilers, such as those in Java Virtual Machines or PyTorch's \gls{jit} backend, monitor program behavior and apply optimizations based on observed execution patterns. Dynamic optimization can exploit runtime information, such as branch frequencies, hot paths, and actual data values, that is unavailable at compile time. The trade-off is that the optimization itself consumes runtime resources.

Our \gls{rl}-based auto-scheduler operates as a static optimizer: it selects transformations before execution, guided by features extracted from the \gls{mlir} code. However, the execution-time feedback used during training provides a form of dynamic information that shapes the learned policy.

\subsection{Control-Flow and Data-Flow Analysis}

Before applying any transformation, the compiler must analyze the program to determine whether the transformation is legal and to estimate whether it will be beneficial. Two families of analysis form the foundation of most compiler optimizations.

\textbf{Control-flow analysis} constructs a representation of the possible execution paths through the program. The most common representation is the \gls{cfg}, where nodes represent \textit{basic blocks}, maximal sequences of instructions with a single entry and a single exit, and edges represent possible transfers of control (branches, jumps, fall-through). The \gls{cfg} enables the compiler to reason about dominance (which blocks must execute before others), loops and reachability.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\textwidth]{ressources/chapter1/control_flow_graph.png}
    \caption{Example of a control-flow graph for a nested loop structure}
    \label{fig:cfg-example}
\end{figure}

\textbf{Data-flow analysis} determines how values are defined, used, and propagated through the program. Key data-flow problems include:

\begin{itemize}
    \item \textbf{Reaching definitions}: For each program point, which assignments to a variable may have reached that point without being overwritten? This analysis supports constant propagation and common subexpression elimination.
    \item \textbf{Live variable analysis}: For each program point, which variables hold values that may be read before being overwritten? This analysis supports register allocation and dead code elimination.
    \item \textbf{Available expressions}: Which expressions have already been computed and not invalidated? This supports global common subexpression elimination.
\end{itemize}

These analyses are typically formulated as iterative fixed-point computations over a lattice of data-flow facts. In the context of \gls{mlir} and our work, similar analyses, such as dependence analysis for loop transformations, are performed using the polyhedral model or by the \gls{mlir} analysis infrastructure.

\section{Loop Nest Optimization}

Loops account for the vast majority of execution time in compute-intensive programs, particularly in scientific computing and machine learning workloads. A \textit{loop nest} is a set of loops where one loop is contained within the body of another. For example, matrix multiplication involves three nested loops iterating over rows, columns, and the inner product dimension. Optimizing loop nests is therefore one of the most impactful activities a compiler can perform.

This section describes the loop nest transformations that are relevant to our reinforcement learning agent. For each transformation, we explain its mechanism, the conditions under which it is legal and beneficial, and its role in the optimization of \gls{mlir} Linalg operations. The agent's ability to select and parameterize these transformations forms the core of its optimization capability.

\subsection{Parallelization}

Parallelization distributes the iterations of a loop across multiple processing units (cores or threads) so that they execute concurrently. A loop can be parallelized if its iterations are \textit{independent}, that is, no iteration reads from or writes to a memory location that is written by another iteration. Loops that carry no data dependencies across iterations are called \textit{parallel loops}, while those that carry dependencies are called \textit{sequential} or \textit{reduction} loops (the latter being parallelizable if the reduction operation is associative and commutative).

In \gls{mlir}, parallelization is expressed through the \texttt{scf.forall} operation, which explicitly marks a loop as parallel and allows the compiler to map its iterations to hardware threads. A related concept is \textit{tiled parallelization}, which combines tiling with parallelization: the loop nest is first tiled, and the outer tile loops are marked as parallel while the inner loops remain sequential. This approach balances parallelism with cache locality.

For our \gls{rl} agent, the \texttt{TiledParallelization} action (denoted \texttt{TP}) applies parallelization to non-reduction loops and tiling to reduction loops within a Linalg operation. If a fusion producer is also specified, the action becomes \texttt{TiledFusion} (\texttt{TPF}), which fuses the producer into the consumer's parallel loop.

\subsection{Interchange (Permutation)}

Loop interchange swaps the order of two nested loops, changing which loop is outermost and which is innermost. The primary motivation for interchange is to improve memory access patterns. When a matrix is stored in row-major order, an outer loop over rows and inner loop over columns yields contiguous memory accesses and good spatial locality; interchanging them causes strided accesses and poor cache utilization (illustrated in Figure~\ref{fig:loop-interchange}). Interchange is legal only when the data dependencies of the original loop nest are preserved under the new iteration order, verified through dependence analysis.

In our \gls{rl} system, the \texttt{Interchange} action (denoted \texttt{I}) selects a permutation of the loop indices. The number of possible permutations grows factorially with the number of loops, so the agent must learn which reorderings are beneficial for a given operation. The action space supports three modes: pre-enumerated candidate permutations, pointer-based sequential selection, and continuous Lehmer-code decoding.

\subsection{Tiling}

Tiling (also called \textit{blocking}) partitions the iteration space of a loop nest into smaller blocks or \textit{tiles}, which are then executed sequentially, with the iterations within each tile executed as before. The primary purpose is to improve cache utilization: by choosing tile sizes that fit within a cache level, the working set of a tile remains in cache throughout its execution. For a nest of $D$ loops, tiling produces $2D$ loops: each of the original $D$ loops contributes an outer tile loop and an inner point loop.

The choice of tile size is critical: tiles that are too large do not fit in cache and provide no benefit; tiles that are too small incur excessive loop overhead and underutilize cache lines. The optimal tile size depends on the cache capacity, the size of the data elements, and the number of arrays accessed in the loop body. In our hardware-aware observation (V1), the agent receives cache size information to inform this choice.

In our system, the \texttt{Tiling} action (denoted \texttt{T}) selects, for each loop in the nest, a tile size from a set of powers of two. A tile size of zero indicates that the loop should not be tiled. While tiling in general does not require the tile size to divide the loop bound (remainder iterations can be handled by an epilogue loop), the current \gls{mlir} \gls{rl} implementation requires this divisibility condition as a simplification; this is among the known limitations of the system.

\subsection{Fusion}

Loop fusion (also called \textit{loop jamming}) merges two or more loops that iterate over the same range into a single loop whose body contains the bodies of all the original loops. The primary benefit is the elimination of intermediate data: when a producer loop writes to a temporary array and a consumer loop reads from it, fusing them replaces the array with a scalar or register-level value, reducing memory traffic and improving locality.

Fusion is legal when there are no backward dependencies between the loop bodies (i.e., the consumer does not depend on a value that the producer computes in a later iteration). In the context of \gls{mlir}, fusion is applied at the operation level: two Linalg operations that are connected by a producer-consumer relationship can be fused into a single operation with a combined loop nest.

Fusion is a key transformation in our system. However, as demonstrated by the case study of the previous team (described in the General Introduction), fusion is not always beneficial: fusing two operations can prevent the individual application of more impactful transformations (such as aggressive parallelization) to each operation. The agent must learn when fusion improves performance and when it hinders other optimizations.

\subsection{Vectorization}

Vectorization transforms a scalar loop that processes one data element per iteration into a vector loop that processes multiple elements simultaneously using \gls{simd} instructions. Modern CPUs support \gls{simd} instruction sets such as SSE (128-bit), AVX2 (256-bit), and \gls{avx}-512 (512-bit), which can operate on 2, 4, 8, or more data elements in a single instruction.

For a loop to be vectorizable, its iterations must be independent (no loop-carried dependencies) and the memory accesses must be contiguous or have a compile-time-known stride. In \gls{mlir}, vectorization is applied to Linalg operations through the vector dialect, which progressively lowers Linalg operations to vector operations and ultimately to \gls{simd} machine instructions.

In our \gls{rl} agent, the \texttt{Vectorization} action (denoted \texttt{V}) is a terminal action: once vectorization is applied to an operation, no further transformations can be applied to it, and the schedule for that operation is considered complete. The agent also supports automatic vectorization preprocessing for convolution and pooling operations.

\section{Challenges in Modern Compiler Optimization}

Despite decades of research and engineering, several fundamental challenges make compiler optimization a difficult, open-ended problem.

\subsection{The Phase Ordering Problem}

The phase ordering problem is the challenge of determining the optimal sequence in which to apply compiler transformations. Transformations are not commutative: applying transformation $A$ before transformation $B$ can produce a different outcome than applying $B$ before $A$, because each transformation modifies the program in ways that affect the applicability and benefit of subsequent transformations.

For a concrete example, consider loop fusion and parallelization applied to a sequence of operations. If fusion is applied first, the two operations are merged, and the resulting combined loop can be parallelized. If parallelization is applied first, each operation is parallelized independently, and fusion may no longer be possible because the parallel loop structure differs. The choice between these two orders depends on the specific operation sizes, the cache hierarchy, and the available parallelism, there is no universally optimal order.

The search space of possible transformation sequences is combinatorial in the number of transformations and their parameters. For a loop nest with $D$ loops, there are $D!$ possible interchange permutations, $\mathcal{O}(T^D)$ possible tiling configurations (where $T$ is the number of tile size options), and an exponential number of interleavings between different transformation types. Exhaustive search is infeasible, and heuristics often fail to capture the interaction effects between transformations.

This problem is the primary motivation for using reinforcement learning: an \gls{rl} agent can learn, through experience, which sequences of transformations are likely to be effective for a given program structure, without requiring a hand-crafted model of transformation interactions.

\subsection{Trade-offs}

Compiler optimization involves fundamental trade-offs between competing objectives:

\begin{itemize}
    \item \textbf{Performance vs.\ code size}: Optimizations such as loop unrolling and function inlining improve execution speed but increase the size of the generated code. On embedded systems with limited memory, code size may be the primary constraint.
    \item \textbf{Performance vs.\ compilation time}: More aggressive optimization analyses and search procedures take longer. For interactive development, fast compilation is preferred; for production deployment, longer optimization passes are acceptable.
    \item \textbf{Generality vs.\ specialization}: Optimizations tailored to specific program patterns (e.g., matrix multiplication of particular sizes) can achieve higher performance but do not transfer to new patterns. General-purpose optimizations apply broadly but leave performance on the table.
\end{itemize}

Our work focuses on the performance objective, measured through execution time speedup. The trade-off with compilation time is addressed by caching: once a transformation sequence has been evaluated, its execution time is stored and reused if the same sequence is encountered again, amortizing the compilation cost over training and evaluation runs.

\subsection{Modern Hardware Integration}

The diversity and rapid evolution of hardware platforms add another dimension of complexity to compiler optimization. Modern computing environments include heterogeneous systems with CPUs, GPUs, NPUs (Neural Processing Units), and specialized accelerators, each with distinct memory hierarchies, instruction sets, and parallelism models. A transformation sequence that is optimal for an AMD EPYC server processor may be suboptimal for an Intel Xeon or an ARM-based processor, even when the same source program is compiled.

Our hardware-aware observation (V1) is a step toward addressing this challenge: by providing the agent with processor specifications, the learned policy can adapt its optimization decisions to the target platform.

\section{\gls{mlir} Compiler Infrastructure}

The Multi-Level Intermediate Representation (\gls{mlir}) is a compiler infrastructure project within the \gls{llvm} ecosystem, introduced by Lattner et al.~\cite{lattner2021mlir}. \gls{mlir} addresses a key limitation of traditional compiler IRs: the assumption of a single, fixed level of abstraction. Traditional IRs operate at one level (e.g., \gls{llvm} \gls{ir} operates close to machine code, while abstract syntax trees operate close to source code), making it difficult to express transformations that span multiple levels. \gls{mlir} instead supports a \textit{multi-level} \gls{ir}, where different \textit{dialects} define operations, types, and attributes at different levels of abstraction, all coexisting within the same representation.

\subsection{Dialects and Core Concepts}

A dialect in \gls{mlir} is a self-contained namespace of operations, types, and attributes that model a specific domain or abstraction level. \gls{mlir} ships with a set of core dialects that cover common compiler tasks. Table~\ref{tab:mlir-dialects} summarizes the primary dialects utilized throughout our system architecture.

\subsection{\gls{mlir} and Automated Optimization}

Several properties of \gls{mlir} make it well-suited for automated optimization via reinforcement learning:

\begin{enumerate}
    \item \textbf{Explicit loop structure}: Linalg operations explicitly encode the loops, access patterns, and computations of an operation, providing the agent with a complete and structured description of the optimization target.
    \item \textbf{Progressive lowering}: \gls{mlir} supports the gradual lowering of operations through the dialect stack. The agent can apply transformations at the Linalg level, then lower the result through progressively more concrete dialects (Affine, \gls{scf}, Vector, \gls{llvm}) to obtain executable code.
    \item \textbf{Python bindings}: \gls{mlir} provides comprehensive Python bindings that allow the construction, manipulation, and execution of \gls{mlir} modules from Python, the language in which our \gls{rl} agent is implemented. The bindings also expose the transform dialect interpreter, which applies transformation sequences to \gls{mlir} modules programmatically.
    \item \textbf{Extensibility}: New dialects and transformations can be added to \gls{mlir} without modifying the core infrastructure.
\end{enumerate}

\subsection{Execution Pipeline}

To measure the performance impact of a transformation sequence, our system compiles and executes the transformed \gls{mlir} code. The module is processed through a series of lowering passes to the \gls{llvm} dialect, translated to \gls{llvm} \gls{ir}, and \gls{jit}-compiled using \gls{mlir}'s ExecutionEngine at optimization level \texttt{-O3}. Finally, the compiled function is invoked through a timed wrapper to measure execution time. This execution pipeline serves as the primary evaluation mechanism for the reward function, as detailed in Chapter~4.

\section{Conclusion}

This chapter has introduced the fundamentals of compiler code optimization that form the technical foundation of this thesis. We described the three-phase compiler pipeline (front-end, middle-end, back-end), surveyed the classification of optimizations (machine-dependent vs.\ independent, static vs.\ dynamic), and introduced the control-flow and data-flow analyses that enable safe and informed transformations.

The core of the chapter focused on loop nest optimization, detailing the transformations commonly used in compilation: parallelization, interchange, tiling, fusion, vectorization, and the additional transformations (unrolling, padding, packing, skewing, unswitching, inversion, peeling) that extend the scope of compiler optimization. For each transformation, we explained its mechanism, legality conditions, and performance implications.

We then discussed the fundamental challenges that motivate the automation of compiler optimization: the phase ordering problem, the trade-offs between competing objectives, and the complexity introduced by diverse and evolving hardware platforms. These challenges explain why hand-crafted heuristics are insufficient and why a learning-based approach is attractive.

Finally, we introduced \gls{mlir}, the compiler infrastructure on which our work is built. \gls{mlir}'s multi-level \gls{ir}, dialect system, explicit operation structure, and Python bindings make it uniquely suitable as a platform for \gls{rl}-based optimization. The Linalg dialect, in particular, provides the structured representation of linear algebra operations that our agent observes and transforms, while the transform dialect provides the mechanism for applying those transformations.

The optimization challenges described in this chapter, especially the phase ordering problem and the vast combinatorial search space of possible transformation sequences, motivate the use of reinforcement learning, which is capable of learning effective strategies through interaction with the environment rather than relying on pre-programmed rules. The next chapter introduces the concepts and algorithms of reinforcement learning that underpin our approach.
