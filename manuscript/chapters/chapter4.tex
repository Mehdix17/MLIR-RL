\section{Motivation}
\label{sec:problem-statement-contributions}

\subsection{Overview of the Problem Statement}
Modern compilers rely on rigid, hand-crafted heuristics to optimize code, which often fail to unlock the full computational potential of underlying hardware for complex deep learning workloads. While formulating code optimization as a Markov Decision Process (MDP) and leveraging deep reinforcement learning (DRL)---as initiated in early baseline models (Bendib~\cite{bendib2024reinforcement}, Tirichine et al.~\cite{tirichine2025reinforcement})---has demonstrated significant promise, scaling these initial proofs-of-concept into robust, generalized mechanisms reveals critical bottlenecks. 

For readers reviewing this chapter independently, the core problem addressed in this thesis is how to scale RL-based compiler optimization so that it can generalize across wildly different algorithmic computation graphs and underlying hardware topologies. Specifically, prior systems are critically crippled by four major limitations:
\begin{itemize}
    \item \textbf{Training Data Sparsity:} Prior agents are trained on narrow, synthetic datasets. They lack exposure to the structural complexity of true deep learning graphs, heavily restricting their ability to optimize realistic, modern tasks.
    \item \textbf{Hardware-Agnostic Policies:} Current agents learn rigid optimization schedules that are completely blind to the underlying hardware architecture. A policy optimized for a lightweight quad-core laptop will aggressively fail when executed on a 256-core HPC node, eliminating cross-hardware portability.
    \item \textbf{Sequential Representation Constraints:} The standard use of recursive \gls{lstm} networks to encode the program state introduces severe informational bottlenecks. LSTMs struggle to maintain context over long-range dependencies inherent in deeply nested loops and broad computation sub-graphs.
    \item \textbf{Sparse Reward Structures:} Extracting a reward only at the end of execution (terminal speedup) yields excessively delayed feedback, causing high variance, slow training convergence, and inefficient exploration spaces.
\end{itemize}

\subsection{Proposed Contributions}
To tackle these fundamental bottlenecks, we propose a completely redesigned, robust RL auto-scheduler capable of generalizing across both complex neural network workloads and diverse CPU platforms. Our primary contributions are:

\begin{itemize}
    \item \textbf{Automated Dataset Generation Pipeline:} We engineered an automated extraction infrastructure with heuristic handlers for dynamic batch sizes. This pipeline successfully sources training sub-graphs from over 20 diverse neural network architectures (including CNNs, Transformers, and GNNs), solving the data sparsity constraint.
    \item \textbf{Hardware-Aware State Representation:} We augmented the agent's observation space with explicit hardware specifications (L1/L2/L3 cache sizes, core counts, and \gls{simd} widths) to enable dynamic, hardware-portable optimization policies.
    \item \textbf{Transformer-Based Loop-Nest Encoding:} We replaced the legacy LSTM networks with a modern Transformer-based encoder. By processing loop levels as tokens and leveraging self-attention, the model accurately untangles the hierarchical dependencies of complex computation graphs.
    \item \textbf{Shaped Multi-Objective Rewards:} We introduced dense, intermediate reward signals driven by static analysis (e.g., arithmetic intensity and vectorizability), drastically accelerating agent convergence by evaluating the mathematical viability of every single transformation step.
\end{itemize}

The remainder of this chapter details the systemic design, theoretical formulation, and practical implementation of these contributions within the \gls{mlir} ecosystem.

\section{Dataset Generation and Evaluation Infrastructure}
To ensure our agent generalizes across modern workloads, we engineered a comprehensive and automated dataset extraction pipeline. Previous approaches relied on manual extraction or heavily hard-coded synthetic generation, severely limiting the structural diversity of the training data. Our automated generation process supports extracting operations from more than 20 different neural network architectures (including vision models, transformer models, BERT variants, and graph neural networks), utilizing different routing mechanisms such as ONNX and \verb|torch-mlir|.

\subsection{Pipeline Overview}
The automated pipeline ingests high-level deep learning models and systematically lowers them into \gls{mlir} representations. It traverses the computation graph, isolates relevant operation sub-graphs, and compiles them to extract profiling baselines. This fully automated flow dynamically translates complex, high-level architectures into executable intermediate representation benchmarks, eliminating the manual bottleneck of earlier systems.

\subsection{Dynamic Batch Size Handling}
During the extraction of operations from diverse neural network models, a major challenge arose concerning dynamic or unresolved batch sizes. Frameworks often export computation graphs with dynamic tensor dimensions, which prevents statically compiled \gls{mlir} benchmarks from successfully executing.

To overcome this, we implemented an automatic dimension handler that resolves and replaces these dynamic batch sizes before compilation. To maximize compatibility, this handler supports multiple configurable strategies:
\begin{itemize}
    \item \textbf{Heuristic Mode:} Infers the optimal batch size based on the dimensions of neighboring tensors within the graph.
    \item \textbf{Statistical Mode:} Assigns batch sizes derived from heavily observed or common execution patterns in the network.
    \item \textbf{Random Mode:} Injects a valid, randomly selected batch size to ensure the extraction sequence completes without interruption.
\end{itemize}

\subsection{Semantic Extraction of Neural Network Blocks}
In contrast to prior work that manually extracted models or utilized completely randomized sequences, our approach automatically extracts neural network blocks directly from real architectures. Critically, this extraction preserves the semantic data-flow relations between producer and consumer operations. 

We imposed a structural limit of a maximum sequence length of 5 operations per benchmark to balance complexity with training feasibility. Furthermore, the pipeline extracts these operations using a sliding spatial shift window of 2. This shifting mechanism ensures an overlapping, dense coverage of the neural network graph, providing the reinforcement learning agent with rich, continuous chains of dependencies.

\subsection{Parallelized Evaluation Pipeline}
To manage the computational overhead of generating baselines and evaluating learned policies on such an extensive dataset, we implemented a parallelized execution pipeline. The framework divides the dataset into smaller, independent chunks that are processed concurrently across the computing cluster. Once execution times and traces are collected for each chunk, a merging step aggregates the results into a unified dataset. This parallelization fundamentally accelerates the iterative research process.

\section{Hardware-Aware Observation}
Prior systems operated without visibility into the underlying hardware, forcing the policy to implicitly memorize hardware-specific heuristics. This blindness critically limits cross-platform portability; a policy that has optimally learned scheduling permutations for a quad-core laptop processor will generate highly suboptimal execution paths when deployed on a 64-core enterprise server, primarily due to catastrophic cache miss rates and thread under-utilization.

To resolve this bottleneck, we inject explicit hardware specifications directly into the observation vector, establishing a hardware-aware state representation. The augmented structural state now natively perceives caching hierarchies, threading limits, and vector capabilities. Specifically, the observation is concatenated with normalizing values for:
\begin{itemize}
    \item \textbf{Cache Hierarchy:} The sizes of the L1, L2, and L3 caches (measured in kilobytes), dictating feasible upper bounds for loop tiling to maintain data locality.
    \item \textbf{Computational Capacity:} The precise number of physical and logical processing cores available.
    \item \textbf{Vector Instruction Potential:} The \gls{simd} vector width (e.g., 256 bits for AVX2, or 512 bits for \gls{avx}-512) alongside the processor's nominal clock speed.
\end{itemize}

By grounding the observation in architectural realities, the policy learns to dynamically adjust tile sizes and parallelization strategies based on available cache capacity and threading limits. This effectively decouples the learned policy from the training hardware, promoting wide-ranging cross-hardware generalization.

\section{Transformer Loop-Nest Encoder}
The previous system processed loop features sequentially using a recursive \gls{lstm} encoder. While effective for simple local dependencies, recurrent sequential architectures fundamentally struggle to model multi-directional, long-range interactions within deeply nested loop structures and producer-consumer contexts. Such bottlenecks obscure the relationships between non-adjacent loops, hampering the model's ability to coordinate large-scale transformations.

To overcome the sequential bottleneck, we engineered a Transformer encoder to parse loop nests. Treating each loop level as a distinct token, the model leverages multiple self-attention layers to simultaneously evaluate the strategic importance of the outermost loop (typically targeted for memory tiling) against the innermost loop (targeted for hardware vectorization).

\subsection{Structured Tokenization}
For each observation sample, the encoder constructs an explicit token sequence. The observation is tokenized into specialized segments: a CLS token, a consumer summary token (a dense projection of the full consumer operation feature vector), a producer summary token, and per-loop tokens corresponding to both the consumer and producer operations. These per-loop tokens encapsulate granular, local features such as loop upper bounds, iterator types (e.g., parallel, reduction), load/store memory access coefficients, arithmetic operation counts, and operation-type contextual encodings.

\subsection{Structural Embeddings}
Transformers are permutation-invariant and do not inherently process sequence order. To maintain the rigorous hierarchical context of the loop nest and the computational data-flow, additive structural embeddings are applied to each token vector before the attention layers. These structural cues include:
\begin{itemize}
    \item \textbf{Role Embedding:} Dictates the specific origin space of the token (global context, consumer operation, or producer operation).
    \item \textbf{Token-Type Embedding:} Delineates whether the token operates as a CLS metric, a summary node, a loop tier, or historical action context.
    \item \textbf{Depth Embedding:} Explicitly denotes the depth index of the loop within the nest hierarchy, providing the physical positional coordinate to the attention heads.
\end{itemize}

This structured tokenization mechanism, guided by positional hierarchy, replaces the sequential limitations of the \gls{lstm} with a highly parallelizable, attention-driven backbone that fluidly perceives dependencies across arbitrary looping depths.

\section{Shaped Reward}
In initial iterations, the agent relied exclusively on a sparse, terminal reward structure---specifically, the execution speedup ratio over the unoptimized baseline, granted only at the completion of an entire transformation episode. This sparsity severely hindered early training. Because random exploratory actions rarely align sequentially to produce immediately faster code, the agent was subjected to high variance in gradient estimates, long plateaus before convergence, and inefficient sample utilization.

We mitigate this by introducing dense, intermediate shaped rewards steered by an aggregated static efficiency score. Rather than waiting for a terminal execution pulse, the environment evaluates the localized heuristic quality of each transformation step immediately after it is applied. 

\subsection{Static Efficiency Scoring}
The static efficiency score is computed for every code state permutation by quantifying specific \gls{mlir} operational properties without requiring expensive ahead-of-time compilation. This score blends three pivotal heuristics:
\begin{itemize}
    \item \textbf{Arithmetic Intensity:} An estimation derived from the ratio of floating-point operations relative to the bytes of memory moved. Higher arithmetic intensity implies compute-boundedness, prioritizing parallelization or vectorization over memory tiling.
    \item \textbf{Vectorizability Status:} A discrete signal determining if the structural requirements for successful vectorization are met (e.g., floating-point bounds with sufficient operand quantities).
    \item \textbf{Parallel Loop Ratio:} The proportion of nesting depths that possess parallelizable iterator properties versus sequential reduction hierarchies.
\end{itemize}

\subsection{Interpolating Reward Feedback}
A per-step shaped reward is calculated strictly as the delta (difference) in the static efficiency score before and after an action is applied. This delta evaluates if an action mathematically improved the code's inherent potential. By subsequently blending this continuous, localized intermediate feedback with the terminal ground-truth execution reward, the agent accurately discerns the localized impact of its individual decisions, leading to a markedly faster and more stable policy convergence.

\section{Integrated Combined Architecture}
Our final system formulation fuses the aforementioned modular enhancements into a singular, unified optimization engine. The isolated enhancements individually address distinct layers of the reinforcement learning paradigm: the Observation Space (Hardware Awareness), the Reinforcement Signal (Shaped Rewards), and the Deep Neural Architecture (Transformer Encoder). 

Combining these modules yields potent synergistic properties. The highly expressive Transformer encoder is able to learn complex hierarchical program embeddings, which are then structurally bounded and guided by the injection of the hardware capability limits. Concurrently, the intermediate shaped reward pipeline provides the dense, step-by-step mathematical gradients that are rigorously required to stabilize and converge the heavily parameterized Transformer network from random state initialization. This integrated architecture forms a highly configurable pipeline accessible via native \gls{mlir} Python bindings, managing dataset orchestration and PyTorch learning loops seamlessly across slurm-managed high-performance computing clusters.

\section{Implementation}
\label{sec:implementation}

This section describes the technical tools, software frameworks, and computing resources that underpin the development, training, and evaluation of our reinforcement learning agent.

\subsection{Tools and Frameworks}

The implementation of our system relies on a stack of open-source tools and libraries that span deep learning, compiler infrastructure, experiment tracking, and job orchestration. \\ 

\textbf{Python} \\ 
Python is the primary implementation language of our framework. Its extensive ecosystem of scientific computing and machine learning libraries, combined with its accessibility and rapid prototyping capabilities, makes it well suited for reinforcement learning research. All agent logic, environment interaction, training loops, and evaluation pipelines are implemented in Python~3.11. \\ 

\textbf{PyTorch} \\
PyTorch~\cite{paszke2019pytorch} is the deep learning framework used to construct and train all neural network components of our system, including the policy and value networks of the \gls{ppo} agent and the \gls{lstm} and Transformer encoders. PyTorch provides dynamic computation graphs, automatic differentiation via its autograd engine, and GPU acceleration. In our work, all training and inference runs on CPU nodes, as the primary computational bottleneck is the compilation and execution of \gls{mlir} code rather than neural network forward and backward passes. \\ 

\textbf{MLIR} \\
The \gls{mlir} compiler infrastructure~\cite{lattner2021mlir} is the foundation on which our optimization environment operates. The \gls{mlir} framework provides the Linalg dialect for representing structured linear algebra operations, the transform dialect for encoding transformation schedules, and the progressive lowering pipeline that translates high-level operations into executable machine code. The \gls{mlir} tree used in this work is built from source at the \texttt{release/19.x} branch, enabling direct access to the latest compiler passes and Python bindings. \\ 

\textbf{ONNX} \\
The Open Neural Network Exchange (ONNX)\footnote{\url{https://onnx.ai}} is an open format for representing machine learning models. We use ONNX as one of the routing mechanisms in our dataset generation pipeline (Section~4.2) to export neural network architectures from PyTorch into a standardized graph representation before lowering them to \gls{mlir}. The ONNX intermediate representation preserves the computational graph structure and operator semantics, enabling systematic extraction of operation sub-graphs across diverse model families. \\ 

\textbf{Streamlit} \\
Streamlit\footnote{\url{https://streamlit.io}} is an open-source Python framework for building interactive data applications. We developed a custom evaluation dashboard using Streamlit that visualizes the performance of our \gls{rl} agents against multiple baselines. The dashboard loads evaluation results from the standard results directory structure, parses benchmark metadata (model family, operation type, batch size), and renders interactive charts using Plotly. Key views include execution time comparisons across implementations and PyTorch baselines (eager, compile, and \gls{jit} modes), cumulative speedup distribution curves, per-model-family speedup heatmaps, training convergence plots, transformation schedule inspection, head-to-head statistical comparisons with standard deviation error bars, and correlation analysis. The dashboard supports filtering by benchmark category, model family, operation type, and batch size, enabling rapid qualitative assessment across the four agent versions and the baseline. Results can be exported in CSV and JSON formats for further analysis. \\ 

\textbf{SLURM} \\
The Simple Linux Utility for Resource Management (SLURM)~\cite{yoo2003slurm} is the workload manager that orchestrates job submission, scheduling, and execution on the computing cluster. Training and evaluation jobs are submitted as SLURM batch scripts that handle environment activation (Conda environment, \texttt{.env} variable sourcing), dataset splitting, model checkpointing, and log management. SLURM array jobs enable parallel execution of multiple implementation versions within a single submission. \\

\subsection{MLIR Python Bindings}

A critical component of our implementation is the bridge between Python and the \gls{mlir} C++ compiler infrastructure. The \gls{mlir} project provides comprehensive Python bindings~\cite{lattner2021mlir} that expose the full \gls{mlir} API to Python code. These bindings are generated from the C++ API using a combination of SWIG and pybind11-style wrappers, and they ship as part of the in-tree \gls{mlir} build.

In our workflow, the Python bindings serve three essential functions. First, they enable the construction and manipulation of \gls{mlir} modules from Python: the agent loads benchmark programs as \gls{mlir} text, parses them into in-memory \gls{mlir} modules, and progressively applies transformations using the transform dialect interpreter. Second, they provide the execution engine: the \texttt{mlir.execution\_engine} module \gls{jit}-compiles the lowered \gls{llvm} dialect code to native machine code and exposes a callable function that operates on input and output memory buffers. Third, they expose the C++ diagnostic infrastructure, allowing Python code to catch and handle compilation errors gracefully---an essential capability given that the agent may attempt transformation sequences that are semantically invalid or crash during lowering.

The bindings are accessed by setting \texttt{PYTHONPATH} to the build directory of the compiled \gls{mlir} tree. This path, along with the \texttt{LD\_LIBRARY\_PATH} to the Conda environment and a GCC-14 \texttt{libstdc++} required by the cluster, is configured through a \texttt{.env} file sourced before any script runs.

\subsection{Computing Infrastructure}

All experiments in this work were conducted on the Jubail High-Performance Computing (HPC) cluster at New York University Abu Dhabi (NYUAD). Jubail is a medium-scale Linux-based HPC system launched in March 2022 and hosted at the NYUAD Data Center. The cluster comprises 269 CPU nodes totaling approximately 33,984 CPU cores, interconnected via a high-speed InfiniBand network.

Each CPU node is equipped with two AMD EPYC 7742 processors (64 cores each, 2.25 GHz base clock), providing 128 physical cores and 512 GB of RAM per node. The EPYC 7742 features a 256 KB L1 cache per core, 512 KB L2 cache per core, and a shared 256 MB L3 cache per socket. All training and evaluation in our work was performed on individual CPU nodes allocated exclusively to each job, ensuring no interference from co-located workloads on execution time measurements.

Jobs are submitted to the cluster via the SLURM workload manager. The Conda environment located at \texttt{\textasciitilde/envs/mlir} provides Python~3.11 and all required dependencies, including PyTorch, NumPy, Plotly, and Streamlit. The \texttt{.env} file configures the paths to the \gls{mlir} build artifacts, shared libraries, and external C++ tools (AST dumper, vectorizer) used during dataset generation and baseline extraction.

\section{Conclusion}
The architectural refinements detailed in this chapter successfully adapt \gls{rl} for the complexities of modern compiler construction. By leveraging parallel dataset extraction spanning diverse deep learning families, integrating strict hardware-bound limits within the observation, deploying a hierarchical transformer encoder, and introducing localized shaped rewards, the framework establishes a robust optimization foundation. We also described the implementation details---the tools and frameworks (PyTorch, \gls{mlir}, Streamlit, SLURM), the \gls{mlir} Python bindings that bridge high-level agent logic with low-level compilation,the Jubail HPC computing infrastructures---that make the system practical at scale. Working in tandem, these modules navigate the expansive combinatorial search space of loop transformations precisely and stably. Exploring how these innovations translate theoretical improvements into pragmatic, empirical execution speedups---and detailing the rigorous evaluation protocols applied to benchmarking their true utility---is the primary focus of the succeeding chapter, \textit{Experiments and Results}.
