\section{Motivation}
\label{sec:problem-statement-contributions}

\subsection{Overview of the Problem Statement}
Modern compilers rely on rigid, hand-crafted heuristics to optimize code. These heuristics fail to adapt across the growing diversity of hardware architectures and deep learning workloads. While formulating code optimization as an \gls{mdp} and leveraging \gls{drl}---as initiated by Bendib et al.~\cite{bendib2024reinforcement} and extended by Tirichine et al.~\cite{tirichine2025reinforcement}---has demonstrated feasibility, scaling these initial proofs-of-concept to handle realistic neural network programs reveals four limitations:

\begin{itemize}
    \item \textbf{Training Data Sparsity:} Prior agents trained on narrow, synthetic datasets. They lack exposure to the structural diversity of real deep learning computation graphs, limiting their ability to optimize the operations found in modern neural network architectures.
    \item \textbf{Hardware-Agnostic Policies:} Existing agents learn optimization schedules without awareness of the target hardware. A policy trained on one processor cannot adapt its tile sizes or parallelization decisions when deployed on a different \gls{cpu}, eliminating cross-hardware portability.
    \item \textbf{Sequential Representation Constraints:} The recursive \gls{lstm} networks used by prior work to encode program state process loop levels sequentially. This sequential structure causes information about early loop levels to decay by the time later levels are processed, weakening the representation of deep loop nests where relationships between non-adjacent levels determine optimization quality.
    \item \textbf{Unreliable Reward Shaping:} Prior \gls{mlir}-\gls{rl} systems rely on a sparse terminal reward received only at episode completion.
\end{itemize}

\newpage
\subsection{Proposed Contributions}
To address these limitations, we extend the prior \gls{mlir}-\gls{rl} framework with four architectural enhancements:

\begin{itemize}
    \item \textbf{Automated Dataset Generation Pipeline:} An extraction infrastructure that sources training benchmarks from 18 neural network architectures spanning 8 categories, with heuristic handlers for dynamic batch sizes. This expands the training data beyond the narrow synthetic benchmarks used in prior work.
    \item \textbf{Hardware-Aware State Representation:} A 7-element hardware feature vector (L1/L2/L3 cache sizes, physical and logical core counts, \gls{simd} width, and clock frequency) concatenated to the policy and value network inputs. This enables the agent to condition tile sizes and parallelization decisions on the target hardware.
    \item \textbf{Transformer-Based Loop-Nest Encoder:} A self-attention-based encoder that replaces the sequential \gls{lstm} architecture of prior work. Loop levels are tokenized and processed simultaneously, allowing direct attention between non-adjacent levels.
    \item \textbf{Shaped Reward Exploration:} An investigation of dense intermediate reward signals based on static code analysis---arithmetic intensity, vectorizability, and parallel-loop ratio---computed at each transformation step.
\end{itemize}

The following sections detail the implementation of these contributions within the \gls{mlir} ecosystem.

\newpage
\subsection{Pipeline Overview}
The automated pipeline ingests high-level deep learning models and systematically lowers them into \gls{mlir} Linalg representations. The conversion proceeds through a standardized path: each PyTorch model is exported to \gls{onnx} via \texttt{torch.onnx.export}, then lowered to the Linalg-on-tensors dialect through the \texttt{torch-mlir-import-onnx} and \texttt{torch-mlir-opt} tools. This fully automated flow translates complex, high-level architectures into executable intermediate representation benchmarks, eliminating the manual bottleneck of earlier systems. For autoregressive decoder models (GPT-2 and Llama~3.2), a compatibility wrapper resolves \texttt{DynamicCache} tracing issues by passing cache positions explicitly.

\subsection{Dynamic Batch Size Handling}
During the extraction of operations from diverse neural network models, a major challenge arose concerning dynamic or unresolved batch sizes. Frameworks often export computation graphs with dynamic tensor dimensions, which prevents statically compiled \gls{mlir} benchmarks from successfully executing.

To overcome this, we implemented an automatic dimension handler that resolves and replaces these dynamic batch sizes before compilation. To maximize compatibility, this handler supports multiple configurable strategies:
\begin{itemize}
    \item \textbf{Heuristic Mode:} Infers the optimal batch size based on the dimensions of neighboring tensors within the graph.
    \item \textbf{Statistical Mode:} Assigns batch sizes derived from heavily observed or common execution patterns in the network.
    \item \textbf{Random Mode:} Injects a valid, randomly selected batch size to ensure the extraction sequence completes without interruption.
\end{itemize}

\subsection{Single-Operation Extraction}

The pipeline extracts isolated Linalg operations from raw model files. Each output file wraps a single operation in a timed \texttt{@main} function with \texttt{@nanoTime()} calls, producing self-contained executable benchmarks. The extraction retains all Linalg operations from every model, including element-wise \texttt{linalg.generic} operations, to ensure comprehensive coverage of the structural diversity present in real-world neural network computation graphs. The final single-operation pool combines our 382 extracted benchmarks with 1335 synthetic benchmarks from previous team dataset \cite{tirichine2025reinforcement}, yielding a total of 1{,}517 benchmarks.

\newpage
\subsection{Block Extraction from Computation Graphs}

To capture producer-consumer relationships, the pipeline extracts operation blocks by traversing the computation graph along data-flow paths. A sliding window of five operations with stride three is applied along each consumer-to-producer chain, producing overlapping sub-graphs with continuous dependency chains. Each extracted block is a self-contained timed benchmark that preserves the semantic data-flow relations between producer and consumer operations. The extraction process produces 3{,}131 blocks in total. Combined with legacy 3{,}736 synthetic blocks, all blocks are assigned entirely to the training set.

\section{Hardware-Aware Observation}
Prior systems operated without visibility into the underlying hardware, forcing the policy to implicitly memorize hardware-specific heuristics. This blindness limits cross-platform portability; a policy that has optimally learned scheduling permutations for a quad-core laptop processor will generate suboptimal execution paths when deployed on a 64-core enterprise server, primarily due to high cache miss rates and thread under-utilization.

To resolve this bottleneck, we inject explicit hardware specifications directly into the observation vector, establishing a hardware-aware state representation. A fixed vector of seven normalized features is concatenated to the embedding output before it feeds into the policy and value networks. Table~\ref{tab:hw-features} lists each feature, its source, and the normalization constant applied.

\begin{table}[H]
    \centering
    \caption{The seven hardware features fed to the observation vector.}
    \label{tab:hw-features}
    \vspace{0.2cm}
    \begin{tabular}{l l}
        \toprule
        \textbf{Feature} & \textbf{Normalization} \\
        \midrule
        L1 data cache (KB) & $\div$ 256 \\
        L2 cache (KB) & $\div$ 4{,}096 \\
        L3 cache (KB) & $\div$ 65{,}536 \\
        Physical cores & $\div$ 256 \\
        Logical cores & $\div$ 512 \\

        \bottomrule
    \end{tabular}

Each feature is extracted from information exposed by the operating system (e.g., cache sizes from \texttt{/sys/devices/system/cpu} and core counts from \texttt{/proc/cpuinfo}). The normalization divisors map each feature to roughly the $[0, 1]$ interval. Features are auto-detected at module import time and can be manually overridden via configuration fields. Setting all fields to zero produces a zero vector, enabling a hardware-agnostic ablation.

By grounding the observation in architectural realities, the policy learns to dynamically adjust tile sizes and parallelization strategies based on available cache capacity and threading limits. This effectively decouples the learned policy from the training hardware, promoting wide-ranging cross-hardware generalization.

\section{Transformer Loop-Nest Encoder}
The previous system processed loop features sequentially using a recursive \gls{lstm} encoder. While effective for simple local dependencies, recurrent sequential architectures fundamentally struggle to model multi-directional, long-range interactions within deeply nested loop structures and producer-consumer contexts. Such bottlenecks obscure the relationships between non-adjacent loops, hampering the model's ability to coordinate large-scale transformations.

To overcome the sequential bottleneck, we engineered a Transformer encoder to parse loop nests. Treating each loop level as a distinct token, the model leverages multiple self-attention layers to simultaneously evaluate the strategic importance of the outermost loop (typically targeted for memory tiling) against the innermost loop (targeted for hardware vectorization). The transformer encoder in the proposed agent employs a model with $d_\text{model} = 256$, $h = 8$ attention heads, $N = 3$ encoder layers, feedforward dimension $d_\text{ff} = 1024$, dropout of $0.1$, and \gls{gelu} activation.

\subsection{Structured Tokenization}
For each observation sample, the encoder constructs a sequence of 17 tokens: a learned \texttt{[CLS]} token initialized from $\mathcal{N}(0, 0.02)$, a consumer summary token, a producer summary token, and up to 7 consumer loop tokens and 7 producer loop tokens (determined by the configurable \texttt{max\_num\_loops} parameter). Table~\ref{tab:transformer-tokens} details the token composition.

    \label{tab:transformer-tokens}
    \vspace{0.2cm}
    \begin{tabularx}{\textwidth}{l c l l X}
        \toprule
        \textbf{Token} & \textbf{Count} & \textbf{Role} & \textbf{Type} & \textbf{Source} \\

The summary tokens project the full 711-dimensional operation feature vector through a two-layer MLP. The per-loop tokens project a 111-dimensional raw loop feature vector (loop bounds, iterator types, access coefficients, operation-type one-hot, and arithmetic counts) through a separate two-layer MLP. Unused token positions are masked, preventing the attention mechanism from attending to absent loops. When no producer exists, the producer tokens are zero vectors.

\subsection{Structural Embeddings}
Transformers are permutation-invariant and do not inherently process sequence order. To preserve the hierarchical context of the loop nest and the computational data-flow, three learned additive embeddings are applied to each token vector before the attention layers. Table~\ref{tab:structural-embeddings} details their vocabulary sizes.

    \centering

    \label{tab:structural-embeddings}

        \toprule
        \textbf{Embedding} & \textbf{Vocabulary} & \textbf{Description} \\
        \midrule

        \bottomrule

The role embedding distinguishes which part of the computation graph each token originates from. The token-type embedding delineates the semantic category (CLS, summary, loop, or action history). The depth embedding explicitly encodes the nesting level of each loop token, giving the attention heads a positional coordinate within the loop hierarchy. All three embeddings are summed with the token vector, and the resulting sequence is processed by three stacked transformer encoder layers with $h = 8$ attention heads, model dimension $d_\text{model} = 256$, feedforward dimension $d_\text{ff} = 1024$, \gls{gelu} activation, dropout of $0.1$, layer normalization applied before attention and feedforward sub-layers (\texttt{norm\_first}), and batch-first ordering.

After the transformer encoder, the output corresponding to the \texttt{[CLS]} token is taken as the pooled representation (CLS pooling). The action history---a multi-hot encoding of the five most recent transformation steps, totaling 1{,}085 dimensions ($5 \times 7 \times 8 \times 3$ for three tiling variant histories plus $5 \times 7 \times 7$ for interchange history)---is concatenated with the pooled CLS vector. The seven-element hardware feature vector is then concatenated, producing a final embedding of $256 + 1{,}085 + 7 = 1{,}348$ dimensions. This embedding is passed to the policy and value networks, each consisting of three fully connected layers with 512 neurons and \gls{relu} activation, followed by the action-specific output heads.

\section{Shaped Reward}
In initial iterations, the agent relied exclusively on a sparse, terminal reward structure---the execution speedup over the unoptimized baseline, granted only at episode completion. This sparsity hindered early training: because random exploratory actions rarely align to produce measurably faster code, the agent received no learning signal for the majority of its transformation steps. This produces high variance in gradient estimates, long plateaus before convergence, and inefficient sample utilization: the agent must complete many episodes simply to discover which actions move the state toward better schedules.

We mitigate this by introducing dense, intermediate shaped rewards computed after every transformation step without executing the code. The core idea is to evaluate, using only static analysis of the \gls{mlir} operation's features, whether a transformation step improves the code's \textit{optimization potential}---its capacity to be further optimized by subsequent steps---rather than requiring a full compilation and execution cycle.

\subsection{Design Principle: Rewarding Potential, Not Outcome}

At step $t$, the agent applies an action $a_t$ that transforms the operation features from $f_{t-1}$ to $f_t$. A shaped reward based on the absolute features $f_t$ would not distinguish between a state that is inherently promising (e.g., a large matmul with many parallelizable loops) and one that the agent actively improved. To reward the agent's \textit{decisions}, we compute the \textit{delta}---the change in a static efficiency score $S(f)$ before and after the action. If an action increases the score, the agent receives positive feedback; if it decreases the score, it receives negative feedback. This approach directly addresses the credit assignment problem that plagues sparse terminal rewards: each action is evaluated on its immediate contribution to the code's optimization potential.

\subsection{Static Efficiency Score: Three Complementary Heuristics}

The static efficiency score $S(f)$ combines three heuristics, each weighted to reflect its importance:

\begin{equation}
    S(f) = w_\text{ai} \cdot \log_{10}\!\bigl(1 + \text{AI}(f)\bigr) \;+\; w_\text{vec} \cdot \mathbb{I}_\text{vec}(f) \;+\; w_\text{par} \cdot \frac{N_\text{parallel}}{\max(1, N_\text{total})}
    \label{eq:efficiency-score}
\end{equation}

\textbf{Arithmetic intensity} ($\text{AI}(f)$, weight $w_\text{ai} = 1.0$) is the ratio of floating-point operations to bytes transferred. We estimate it from the iteration space volume (the product of all loop upper bounds) multiplied by the number of arithmetic operations per iteration, divided by the total bytes accessed across all load and store patterns. To handle the wide range of values (from $< 1$ for element-wise operations to $> 100$ for large matrix multiplications), we apply $\log_{10}$ compression, which also keeps the signal in the same numerical regime as the terminal $\log_{10}$ speedup reward. Higher arithmetic intensity generally indicates better cache reuse and lower memory bandwidth pressure.

\textbf{Vectorizability} ($\mathbb{I}_\text{vec}(f)$, weight $w_\text{vec} = 0.1$) is a binary flag that equals $1.0$ when the operation satisfies \gls{mlir}'s vectorization preconditions. A transformation that turns a non-vectorizable operation into a vectorizable one receives a positive delta. The weight is modest because vectorizability is a necessary but not sufficient condition for speedup.

\textbf{Parallel-loop ratio} ($N_\text{parallel} / N_\text{total}$, weight $w_\text{par} = 0.1$) is the fraction of loops without loop-carried dependencies. Transformations that expose parallelism (e.g., interchanging a reduction loop to an outer position) increase this ratio. Like vectorizability, the weight is modest because actual parallel speedup depends on factors captured only by the terminal execution reward.

\subsection{Per-Step Shaped Reward}

The per-step shaped reward is the scaled change in the efficiency score, clipped to prevent any single step from dominating the episode:

\begin{equation}
    r_t^\text{shaped} = \text{clip}\!\Bigl( \bigl(S(f_t) - S(f_{t-1})\bigr) \times \lambda_\text{scale} \;+\; \lambda_\text{vec\_bonus} \cdot \mathbb{I}[a_t = \text{V}],\; -C,\; +C \Bigr)
    \label{eq:per-step-reward}
\end{equation}

Clipping ($C = 2.0$) stabilizes training by preventing a single transformation that sharply changes one metric from overwhelming the cumulative reward. A small vectorization bonus ($\lambda_\text{vec\_bonus} = 0.2$) offsets the fact that vectorization is a terminal action after which the score cannot further improve, counterbalancing the agent's incentive to delay it. \subsection{Terminal Reward and Success-Contingent Negation}

The shaped rewards are accumulated during the trajectory. At episode completion, a terminal reward $R_\text{terminal}$ is computed based on the execution outcome of the final transformed code, and it is added to the last intermediate shaped reward:

\begin{equation}
    R_\text{terminal} = \begin{cases}
        -5.0  & \text{if transformation application fails} \\
        -20.0 & \text{if execution fails (wrong results, timeout)} \\
        \log_{10}\!\left(\frac{T_\text{baseline}}{T_\text{optimized}}\right) & \text{if execution succeeds}
    \end{cases}
    \label{eq:terminal-reward}
\end{equation}

The penalty values are asymmetric by design. A transformation failure ($-5.0$) is a recoverable event: the agent may have selected a valid but poorly-parameterized action. An execution failure ($-20.0$) is more penalized, indicating that the resulting code either produces incorrect results or fails to complete within a timeout window, and the agent should strongly avoid such sequences.

A critical safety mechanism prevents the agent from exploiting the shaped reward in undesirable ways. If the final execution of the transformed code fails, all intermediate shaped rewards accumulated during the episode are zeroed out. Without this mechanism, the agent could learn to maximize static efficiency scores through transformations that ultimately produce code that crashes during lowering or execution, a form of reward hacking where the agent optimizes the proxy metric (the static score) at the expense of the true objective (successful execution). By conditioning intermediate rewards on final execution success, we ensure that the agent is only rewarded for transformation paths that lead to compilable and executable code.

When execution succeeds, the agent receives the full shaped reward trajectory plus the log-speedup terminal reward. This combination provides dense per-step guidance during training while keeping the terminal execution speedup as the ultimate arbiter of schedule quality.

\section{Integrated Combined Architecture}
Our final system combines all the modular improvements into one unified optimization engine. Each improvement targets a specific part of the reinforcement learning pipeline: the Observation Space (Hardware Awareness), the Reward Signal (Shaped Rewards), and the Neural Network (Transformer Encoder).
Combining these modules produces strong synergistic effects. The Transformer encoder learns rich program representations, which are kept within bounds by the hardware constraints. At the same time, the shaped reward pipeline provides frequent, step-by-step feedback signals that help train and stabilize the Transformer network from the start.

\section{Implementation}
\label{sec:implementation}

This section describes the technical tools, software frameworks, and computing resources that underpin the development, training, and evaluation of our reinforcement learning agent.

\subsection{Tools and Frameworks}

The implementation of our system relies on a stack of open-source tools and libraries that span deep learning, compiler infrastructure, experiment tracking, and job orchestration. \\ 

\textbf{Python} \\ 
Python is the primary implementation language of our framework. Its extensive ecosystem of scientific computing and machine learning libraries, combined with its accessibility and rapid prototyping capabilities, makes it well suited for reinforcement learning research. All agent logic, environment interaction, training loops, and evaluation pipelines are implemented in Python~3.11. \\ 

\textbf{PyTorch} \\
PyTorch~\cite{paszke2019pytorch} is the deep learning framework used to construct and train all neural network components of our system, including the policy and value networks of the \gls{ppo} agent and the \gls{lstm} and Transformer encoders. PyTorch provides dynamic computation graphs, automatic differentiation via its autograd engine. \\

\textbf{MLIR} \\
The \gls{mlir} compiler infrastructure~\cite{lattner2021mlir} is the foundation on which our optimization environment operates. The \gls{mlir} framework provides the Linalg dialect for representing structured linear algebra operations, the transform dialect for encoding transformation schedules, and the progressive lowering pipeline that translates high-level operations into executable machine code. The \gls{mlir} tree used in this work is built from source at the \texttt{release/19.x} branch, enabling direct access to the latest compiler passes and Python bindings. \\ 

\gls{onnx} \\
The Open Neural Network Exchange \footnote{\url{https://onnx.ai}} is an open format for representing machine learning models. We use \gls{onnx} as the routing mechanism in our dataset generation pipeline to export neural network architectures from PyTorch into a standardized graph representation before lowering them to \gls{mlir}. The \gls{onnx} intermediate representation preserves the computational graph structure and operator semantics, enabling systematic extraction of operation sub-graphs across diverse model families. \\ 

\textbf{Streamlit} \\
Streamlit\footnote{\url{https://streamlit.io}} is an open-source Python framework for building interactive data applications. We developed a custom evaluation dashboard that visualizes the performance of our \gls{rl} agents against multiple baselines. \\

\gls{slurm} \\
The Simple Linux Utility for Resource Management~\cite{yoo2003slurm} is the workload manager that orchestrates job submission, scheduling, and execution on the computing cluster. Training and evaluation jobs are submitted as \gls{slurm} batch scripts that handle environment activation (Conda environment, \texttt{.env} variable sourcing), dataset splitting, model checkpointing, and log management. \\

\subsection{MLIR Python Bindings}

The \gls{mlir} project provides Python bindings~\cite{lattner2021mlir} that expose the \gls{mlir} API to Python. In our workflow, they serve three functions: constructing and manipulating \gls{mlir} modules from Python via the transform dialect interpreter; \gls{jit}-compiling lowered code through the \texttt{mlir.execution\_engine} module; and catching compilation errors gracefully when the agent attempts invalid transformation sequences. The bindings are accessed by setting \texttt{PYTHONPATH} to the in-tree \gls{mlir} build directory, configured through a \texttt{.env} file sourced before any script runs.

\subsection{Computing Infrastructure}

All experiments in this work were conducted on the Bergamo partition of the \gls{hpc} cluster at New York University Abu Dhabi (NYUAD), unless otherwise specified. Bergamo nodes are equipped with two AMD EPYC 9754 processors (Zen 4, 128 cores each, 2.25~GHz base clock), providing 256 physical cores and 1~TB of RAM per node. The EPYC 9754 features a 32~KB L1 cache per core, 1~MB L2 cache per core, and a shared 256~MB L3 cache per socket.

For the multi-hardware portability experiments (Section~5.4), two additional cluster partitions are used: Jubail, equipped with AMD EPYC 7742 processors (Zen 2, 128 cores, 512~GB RAM, 256~MB L3), and Dalma, equipped with Intel Xeon E5-2680~v4 processors (Broadwell, 28 cores, 112~GB RAM, 35~MB L3). All training and default evaluation runs use individual Bergamo nodes allocated exclusively to each job, ensuring no interference from co-located workloads on execution time measurements.

Jobs are submitted to the cluster via the \gls{slurm} workload manager. The Conda environment located at \texttt{\textasciitilde/envs/mlir} provides Python~3.11 and all required dependencies, including PyTorch, NumPy, Plotly, and Streamlit. The \texttt{.env} file configures the paths to the \gls{mlir} build artifacts, shared libraries, and external C++ tools (\gls{ast} dumper, vectorizer) used during dataset generation and baseline extraction.

\section{Conclusion}
The architectural refinements detailed in this chapter successfully adapt \gls{rl} for the complexities of modern compiler construction. By leveraging parallel dataset extraction spanning diverse deep learning families, integrating strict hardware-bound limits within the observation, deploying a hierarchical transformer encoder, and introducing localized shaped rewards, the framework establishes a robust optimization foundation. We also described the implementation details---the tools and frameworks (PyTorch, \gls{mlir}, Streamlit, \gls{slurm}), the \gls{mlir} Python bindings that bridge high-level agent logic with low-level compilation, the Bergamo \gls{hpc}---that make the system practical at scale. Working in tandem, these modules navigate the expansive combinatorial search space of loop transformations precisely and stably. Exploring how these innovations translate theoretical improvements into pragmatic, empirical execution speedups---and detailing the rigorous evaluation protocols applied to benchmarking their true utility---is the primary focus of the succeeding chapter, \textit{Experiments and Results}.
