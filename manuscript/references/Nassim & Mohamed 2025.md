# **Abstract** {#abstract}

This work focuses on the use of deep reinforcement learning (DRL) to automate code optimization within modern compiler infrastructures. Code optimization is a critical step in program transformation that aims to improve performance and reduce resource consumption while preserving correctness. Traditional approaches to code optimization rely on manual or heuristic-based methods, which are often time-consuming and require deep knowledge of low-level programming and hardware. To overcome these limitations, this project proposes a DRL-based system capable of automatically learning and applying optimization strategies without human intervention. The system targets the MLIR (Multi-Layer Intermediate Rep-resentation) compiler framework and is designed to generalize across different application domains. To evaluate this generality, the system is tested on two distinct domains: neu-ral network operators and Lattice Quantum Chromodynamics (LQCD) simulations. These domains differ significantly in computational patterns and optimization needs. The DRL environment is carefully constructed to represent code states, define legal transformation actions, and incorporate a reward function that reflects the effectiveness of applied optimiza-tions. Additionally, The project includes two new proposed methods: a new architecture inspired from Pointer Networks for the interchange action and an adaptation of the Alp-haZero algorithm for the code optimization problem. The proposed approach demonstrates promising results in learning optimization sequences that improve code performance across varied computational workloads.

**Keywords:** automatic code optimization, deep reinforcement learning, MLIR, neural networks, LQCD

# **General Introduction** {#general-introduction}

The increasing complexity of modern software applications and the heterogeneity of hard-ware architectures have made code optimization a critical concern in computer science. As computational workloads continue to grow in scale and sophistication (ranging from deep learning models to scientific simulations) achieving high performance while maintaining en-ergy efficiency and portability becomes more challenging. Traditional compiler optimization strategies often rely on hand-crafted rules, static cost models, or heuristic scheduling. These techniques typically require domain-specific expertise and are limited by their inability to adapt to new workloads or architectures, especially when the interaction between different optimizations becomes non-trivial.

Recent advancements in [Machine Learning](#ml-machine-learning.) have opened new possibilities for compiler optimization. In particular, [Deep Reinforcement Learning (Deep RL)](#deep-rl-deep-reinforcement-learning.) has emerged as a promising paradigm capable of learning complex sequences of optimization passes that im-prove runtime performance while preserving program semantics. Unlike static approaches, [Deep RL](#deep-rl-deep-reinforcement-learning.)\-based systems can adapt to diverse code patterns and hardware characteristics, of-fering the potential to surpass human-designed heuristics in both performance and generality.

This work explores the question: **How to build a [Deep Reinforcement Learning](#deep-rl-deep-reinforcement-learning.) system that generalizes well across various domains and learns effective and more complex optimization strategies?** Tackling this problem requires addressing several sub-challenges, including designing an expressive yet tractable action space, encoding code struc-ture and semantics into suitable state representations, and defining reward functions that reflect long-term performance gains.

To address these challenges, we propose a [Deep RL](#deep-rl-deep-reinforcement-learning.)\-based optimization system built upon the [Multi-Layer Intermediate Representation (MLIR)](#mlir-multi-layer-intermediate-representation.) compiler infrastructure. [MLIR](#mlir-multi-layer-intermediate-representation.) is designed for modularity and extensibility, providing a unified abstraction for represent-ing diverse computational models and hardware targets. It is particularly well-suited for optimizing compute-intensive workloads such as neural network operators and stencil-based simulations. Its multi-level [IR](#ir-intermediate-representation.) design facilitates high-level domain-specific transformations as well as low-level code generation, making it an ideal substrate for learning-based optimization frameworks.

Our proposed system is domain-independent by design and is empirically evaluated on two distinct domains: neural network operations and [Lattice Quantum ChromoDynamics](#lqcd-lattice-quantum-chromodynamics.)

# **Chapter 1** {#chapter-1}

**Code Optimization**

1. ## **Introduction** {#introduction}

Code optimization is an important program transformation phase that aims to enhance code performance by rewriting intermediate code in order to reduce resource consumption (i.e. [CPU](#cpu-central-processing-unit.) and memory) while maintaining code correctness. Many current domains require heavy computations that would take too long to execute with naive code, which made code optimization as crucial as it is nowadays. Moreover, the increasing complexity of computing architectures such as [CPU](#cpu-central-processing-unit.)s, [GPU](#gpu-graphics-processing-unit.-hpc-high-performance-computer.-if-interleaving-factor.)s and [TPU](#tpu-tensor-processing-unit.)s provides promising opportunities to further optimize code execution.

However, for the code optimization process to be correct and effective it has to improve the overall execution time of the program, preserve the program’s semantics and correctness by applying legal transformations, i.e. the optimized code should yield the same results as the original code, and keep the overall compilation time below a maximum acceptable bound.

Researchers have tried different methods to optimize code. Traditional methods involve manual transformation of the source code to enhance its performance or performing these transformations automatically by building a rule-based agent. However, such methods require knowledge of low-level programming and a deep understanding of the target hardware archi-tecture. Other traditional methods rely on heuristics to apply transformations to approach the most optimized code performance. More recent work relies on deep learning techniques to train an agent that would automatically perform transformations without manual human intervention.

2. ## **Code optimization variants**

Code optimization encompasses a diverse set of strategies applied at different stages of the compilation process and targeting different abstraction levels. These strategies can be broadly categorized based on the stage of application and their dependence on hardware specifics.

1. ### **Optimization levels**

Optimization can be performed at various levels \[[Gong et al., 2025](#[gong-et-al.,-2025]-gong,-j.,-voskanyan,-v.,-brookes,-p.,-wu,-f.,-jie,-w.,-xu,-j.,-giavrimis,-r.,-basios,-m.,-kanthan,-l.,-and-wang,-z.-\(2025\).-language-models-for-code-optimization:-survey,-challenges-and-future-directions.-arxiv-preprint-arxiv:2501.01277.)\], including the source code, [Intermediate Representation (IR)](#ir-intermediate-representation.), and binary levels. At the source code level, modify-ing algorithms, data structures, or implementation details can greatly improve performance \[[Shypula et al., 2023](#[shypula-et-al.,-2023]-shypula,-a.,-madaan,-a.,-zeng,-y.,-alon,-u.,-gardner,-j.,-hashemi,-m.,-neubig,-g.,-ranganathan,-p.,-bastani,-o.,-and-yazdanbakhsh,-a.-\(2023\).-learning-performance-improving-code-edits.-arxiv-preprint-arxiv:2302.07867.)\]. The [IR](#ir-intermediate-representation.) level allows for optimization techniques such as dead code elimination, loop unrolling, and vectorization, which minimize redundant computations and leverage hardware capabilities \[[Voss and Eigemann, 2001](#[voss-and-eigemann,-2001]-voss,-m.-j.-and-eigemann,-r.-\(2001\).-high-level-adaptive-pro-gram-optimization-with-adapt.-in-proceedings-of-the-eighth-acm-sigplan-symposium-on-principles-and-practices-of-parallel-programming,-pages-93–102.)\]. At the binary level, link-time op-timizations, such as instruction scheduling and memory layout adjustments, further enhance computational efficiency and memory access performance \[[Fernandez, 1995](#[fernandez,-1995]-fernandez,-m.-f.-\(1995\).-simple-and-effective-link-time-optimization-of-modula-3-programs.-in-proceedings-of-the-acm-sigplan-1995-conference-on-program-ming-language-design-and-implementation,-pages-103–115.)\].

2. ### **Machine-Dependent vs Machine-Independent Optimization**

Code optimization includes a wide range of possible transformations, some of which can significantly enhance performance, while others may have a negative impact. Therefore, it is crucial to carefully select the transformations to be applied. According to \[[Collard, 2003](#[collard,-2003]-collard,-j.-f.-\(2003\).-reasoning-about-program-transformations:-imperative-programming-and-flow-of-data.-springer.)\], optimizations can generally be classified into two categories: machine-dependent optimization and machine-independent optimization.

#### **Machine-Dependent Optimization**

Machine-dependent optimization tailors the code to the characteristics of a specific hard-ware architecture during compilation to achieve optimal performance. It relies on hardware-specific features to improve efficiency \[[Collard, 2003](#[collard,-2003]-collard,-j.-f.-\(2003\).-reasoning-about-program-transformations:-imperative-programming-and-flow-of-data.-springer.)\].

This type of optimization can be applied at different stages of the compilation process, from the source code to the final machine code. Typically, it involves low-level optimizations that take advantage of specific hardware functionalities. Common techniques include:

* **Architecture-specific compiler directives**: Using compiler hints to optimize exe-cution for a particular processor.

  * **Assembly code optimization**: Fine-tuning low-level instructions for performance gains.

    * **Cache management optimization**: Reducing memory access latency by optimizing data placement in caches.

Machine-dependent optimization is particularly useful for programs targeting a specific hardware platform that require maximum performance.

#### **Machine-Independent Optimization**

Machine-independent optimization is a code optimization approach that aims to improve a program’s performance without considering the specifics of a particular hardware archi-tecture. This type of optimization can also be applied at various stages of the compilation process, from the source code to the generated machine code.  
Common techniques used in machine-independent optimization include:

3. *LOOP NEST OPTIMIZATION*	5

* **Algorithmic optimization**: Modifying the underlying algorithm to improve perfor-mance.

* **Data structure optimization**: Restructuring data for more efficient processing.

* **Loop optimizations**: Techniques such as loop permutation, loop unrolling, and loop fusion.

* **Intermediate variable elimination**: Reducing unnecessary variables to streamline computation.

* **Vectorization**: Leveraging vector instructions for parallel processing.

* **Function call optimization**: Reducing function call overhead through inlining or other techniques.

Machine-independent optimization is particularly valuable for cross-platform programs or applications that need to run on different hardware architectures (e.g., [TPU](#tpu-tensor-processing-unit.)s, [GPU](#gpu-graphics-processing-unit.-hpc-high-performance-computer.-if-interleaving-factor.)s, and [CPU](#cpu-central-processing-unit.)s). By adopting this approach, developers can improve performance across multiple platforms without focusing on the specific features of each processor \[[Collard, 2003](#[collard,-2003]-collard,-j.-f.-\(2003\).-reasoning-about-program-transformations:-imperative-programming-and-flow-of-data.-springer.)\].

#### **Hybrid Optimization**

Choosing between machine-dependent and machine-independent optimization is a com-plex decision shaped by factors such as platform diversity, target users, and performance ex-pectations. Machine-dependent optimizations can yield highly efficient code tailored to a spe-cific hardware architecture but often sacrifice portability. In contrast, machine-independent optimizations favor portability and broad applicability, though they may miss out on lever-aging hardware-specific capabilities for peak performance. In practice, developers frequently adopt a hybrid strategy, combining both approaches to balance performance and portability \[[Liu et al., 2019](#[liu-et-al.,-2019]-liu,-s.,-cui,-y.-z.,-zou,-n.-j.,-zhu,-w.-h.,-zhang,-d.,-and-wu,-w.-g.-\(2019\).-revisiting-the-parallel-strategy-for-doacross-loops.-journal-of-computer-science-and-technology,-34:456–475.)\]. This allows critical parts of the code to benefit from hardware-specific tuning, while the rest remains adaptable across various platforms. The optimal choice ulti-mately depends on the project’s goals, constraints, and the range of environments in which the software is expected to run.

3. ## **Loop Nest Optimization**

In computationally intensive programs, loops and functions are often responsible for most of the execution time \[[Liu et al., 2019](#[liu-et-al.,-2019]-liu,-s.,-cui,-y.-z.,-zou,-n.-j.,-zhu,-w.-h.,-zhang,-d.,-and-wu,-w.-g.-\(2019\).-revisiting-the-parallel-strategy-for-doacross-loops.-journal-of-computer-science-and-technology,-34:456–475.)\]. Therefore, targeting these parts specifically could in most cases drastically increase performance. Following this statement, research focused on optimizing loops, including single loops and nested loops called loop nest, as this process contributes to most of the performance improvement by leveraging the target architecture parallelization capabilities and better cache management. However, loop nest optimization can be challenging due to complex dependencies that can exist between different loop itera-tions, which would make finding the most optimal transformation sequence to apply to loop nests a non-trivial task. In this section, the focus is on loop optimization and the various transformation techniques used to improve loop performance.

6	*CHAPTER 1\.  CODE OPTIMIZATION*

1. ### **Tiling**

Loop tiling is a loop transformation that makes data accessed in tiles (blocks) instead of accessing it in a standard sequential manner, with the tile size being a user-defined parameter. Each loop would be transformed into two loops: an inner loop that would iterate over elements inside the tile and an outer loop that iterates over the tiles. This transformation leverages spatial and temporal locality of data accesses within the loop nest targeting different levels of memory, including cache levels \[[Cardoso et al., 2017](#[cardoso-et-al.,-2017]-cardoso,-j.-m.,-coutinho,-j.-g.-f.,-and-diniz,-p.-c.-\(2017\).-chapter-5---source-code-transformations-and-optimizations.-in-cardoso,-j.-m.,-coutinho,-j.-g.-f.,-and-diniz,-p.-c.,-editors,-embedded-computing-for-high-performance,-pages-137–183.-morgan-kaufmann,-boston.)\]. Moreover, loop tiling helps reduce parallelization overhead by assigning multiple iterations of computation tasks for each thread.    4. ## **MLIR  Compiler**

[Multi-Layer Intermediate Representation (MLIR)](#mlir-multi-layer-intermediate-representation.) \[[Lattner et al., 2020](#[lattner-et-al.,-2020]-lattner,-c.,-amini,-m.,-bondhugula,-u.,-cohen,-a.,-davis,-a.,-pienaar,-j.,-riddle,-r.,-shpeisman,-t.,-vasilache,-n.,-and-zinenko,-o.-\(2020\).-mlir:-a-compiler-infrastructure-for-the-end-of-moore’s-law.-arxiv-preprint-arxiv:2002.11054.)\] is an open-source compiler framework designed to represent, transform, and optimize code across various ab-straction levels. Its name, *Multi-Level Intermediate Representation*, reflects its foundational philosophy of providing a unified, extensible representation that bridges high-level [DSL](#dsl-domain-specific-language.)s and low-level hardware-specific code.

Unlike traditional intermediate representations such as LLVM [IR](#ir-intermediate-representation.), MLIR incorporates principles from polyhedral[1](#1the-polyhedral-model-provides-a-mathematical-representation-of-loop-nests-as-sets-of-iteration-points-within-a-multidimensional-space,-forming-structures-called-polyhedra.-this-model-enables-complex-loop-restructuring-by-expressing-its-techniques-with-fine-grained-elements-such-as-loop-iterations-and-instruction-instances.) loop optimization directly into its core structure. This integra-tion allows MLIR to elegantly express computations involving deep loop nests and high-dimensional dense matrices, making it particularly suited for advanced computational do-mains such as deep learning and scientific computing. At the same time, MLIR remains general enough to handle arbitrary sequential computation, ensuring its applicability across diverse programming paradigms.

A defining advantage of MLIR lies in its ability to represent and progressively optimize computations at multiple abstraction levels. By integrating polyhedral[1](#1the-polyhedral-model-provides-a-mathematical-representation-of-loop-nests-as-sets-of-iteration-points-within-a-multidimensional-space,-forming-structures-called-polyhedra.-this-model-enables-complex-loop-restructuring-by-expressing-its-techniques-with-fine-grained-elements-such-as-loop-iterations-and-instruction-instances.) abstractions with traditional [IR](#ir-intermediate-representation.) concepts, MLIR facilitates powerful loop transformations like loop tiling, in-terchange, skewing, and fusion that are crucial for efficient execution on modern hardware platforms. This flexibility is critical in contemporary computing environments that include diverse hardware architectures such as multicore CPUs, GPUs, and specialized neural net-work accelerators.

Major projects, such as TensorFlow \[[Abadi et al., 2015](#[abadi-et-al.,-2015]-abadi,-m.,-agarwal,-a.,-barham,-p.,-brevdo,-e.,-chen,-z.,-citro,-c.,-corrado,-g.-s.,-davis,-a.,-dean,-j.,-devin,-m.,-ghemawat,-s.,-goodfellow,-i.,-harp,-a.,-irving,-g.,-isard,-m.,-jia,-y.,-jozefowicz,-r.,-kaiser,-l.,-kudlur,-m.,-levenberg,-j.,-man´e,-d.,-monga,-r.,-moore,-s.,-murray,-d.,-olah,-c.,-schuster,-m.,-shlens,-j.,-steiner,-b.,-sutskever,-i.,-talwar,-k.,-tucker,-p.,-vanhoucke,-v.,-vasudevan,-v.,-vi´egas,-f.,-vinyals,-o.,-warden,-p.,-wattenberg,-m.,-wicke,-m.,-yu,-y.,-and-zheng,-x.-\(2015\).-tensorflow:-large-scale-machine-learning-on-heterogeneous-systems.-software-available-from-tensorflow.org.)\] and Mojo \[[Lattner and Davis, 2023](#[lattner-and-davis,-2023]-lattner,-c.-and-davis,-t.-\(2023\).-mojo:-powerful-cpu-and-gpu-programming.-https://www.modular.com/mojo.)\], have adopted MLIR for their compilers and optimization pipelines due to its modular archi-tecture and comprehensive representational capabilities. For example, Google’s TensorFlow leverages MLIR to optimize deep learning models, enabling highly efficient execution on var-ious hardware backends without requiring manual tuning.

In summary, MLIR provides a powerful, flexible, and extensible intermediate represen-tation that effectively bridges high-level abstractions with low-level hardware details. Its innovative approach allows seamless integration of domain-specific optimizations and progres-sive lowering, significantly enhancing compiler flexibility and performance.    8. ### **Passes**

1. ## **The Reinforcement Learning Framework** {#the-reinforcement-learning-framework}

[Reinforcement Learning (RL)](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) is a subfield of [Machine Learning (ML)](#ml-machine-learning.) that focuses on problems involving learning through interaction and exploration to select optimal actions. Unlike supervised learning, which aims to predict hidden patterns and learn input-output mappings from labeled data, [RL](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) involves making sequential decisions to achieve a desired goal by learning through trial and error, guided by the feedback received without explicit instruc-tions or indications of the correct actions. This process enables [RL](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) to tackle dynamic and uncertain problems where optimal behavior must be discovered rather than being predefined.

The reinforcement learning framework fundamentally consists of:

* **Agent:** The decision-making entity being trained, it selects actions based on its ob-servation of the current state.

* **Environment:** The external system with which the agent interacts, it provides a reward and a new state resulting from the agent’s action.

* **States (***S***):** Representations of the environment at a given time, providing the agent with the necessary information to make decisions.

* **Actions (***A***):** Choices made by the agent that affect the environment and cause tran-sitions from one state to another.

* **Rewards (***R***):** Scalar feedback signals (positive or negative) indicating the immediate quality of the action taken, guiding the agent toward learning optimal behavior over time.

These components form the agent-environment interaction cycle of a [Reinforcement Learn-ing](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) system, where the agent interacts with the environment by taking actions, receiving new states and rewards in response, and using this feedback to improve its future decisions (as illustrated in Figure [2.1](#figure-2.1:-reinforcement-learning-interaction-cycle)).  In most real world applications, these interactions continue for

   2. ## **Reinforcement Learning Paradigms** {#reinforcement-learning-paradigms}

   Reinforcement learning systems share several key components \[[Sutton et al., 1998](#[sutton-et-al.,-1998]-sutton,-r.-s.,-barto,-a.-g.,-et-al.-\(1998\).-reinforcement-learning:-an-introduction,-volume-1.-mit-press-cambridge.)\]:

* **Policy (***π***):** Defines the decision making behavior of an agent given a particular state  
  *s*. Formally, it represents a mapping from observations of the environment’s state to actions. More precisely, a policy can be viewed as a function that receives as input the observed state and returns the next action that the agent should take. The choice of action is typically represented as a probability distribution over all possible actions conditioned by a given state, which indicates the likelihood of choosing each action as the next action when the agent observes the state *s*:

*Pπ*(*At* | *St* \= *s*)

where *St* denotes the observed state and *At* denotes the action taken at time-step

*t*. The symbol *π* represents the policy, and this conditional probability is commonly abbreviated as *π*(· | *s*). For a specific action *a*, this becomes:

*π*(*a* | *s*) \= *Pπ*(*At* \= *a* | *St* \= *s*)

* **Return (***Gt***):** Refers to the total accumulated reward after time-step *t* until the end of the episode, usually with discounted future rewards.

*Gt* \= *Rt*\+1 \+ *γRt*\+2 \+ *γ*2*Rt*\+3 · · · \+ *γT* −*t*−1*RT*

where *γ* is the discount factor, and is a value between 0 and 1\. The expected return is the expectation value of the return, denoted by E\[*Gt*\]. Empirically, it is the mean of all the received returns at a time-step *t*.

* **State-value function (***Vπ*(*s*)**):** Estimates the expected return starting from a specific state *s* and following a given policy *π*:

*Vπ*(*s*) \= E*π*\[*Gt* | *St* \= *s*\]

* **Action-value function (***Qπ*(*s, a*)**) (optional):** Estimates the expected return start-ing from a specific state *s*, applying a chosen action *a*, and following a policy *π*:

*Qπ*(*s, a*) \= E*π*\[*Gt* | *St* \= *s, At* \= *a*\]

* **Model of the environment (optional):** Mimics how the environment reacts to different actions, enabling the agent to predict future states and rewards, and that’s done with the help of two (2) types of information:

  * State transition, which is the probability of transitioning to a state *s*′ when the agent is in state *s* and takes action *a*:

*P* (*St*\+1 \= *s*′ | *St* \= *s, At* \= *a*)

Depending on the nature of the problem (such as the state and action space, the level of knowledge available about the environment..) and the type of solution used, different paradigms of reinforcement learning emerge, each leading to the development of distinct methods and approaches:

1. ### **Tabular vs Function Approximation Methods**

The representation of state-value *Vπ*(*s*) and action-value *Qπ*(*s, a*) estimates depends heav-ily on the size of the state space (and the action space, in the case of action-value functions):

2. *REINFORCEMENT LEARNING PARADIGMS*	21

* **Tabular methods:** In environments characterized by small, discrete state and action spaces (such as the faces of a die or a fixed deck of cards) it is practical to explic-itly store value functions in lookup tables, represented as either state values *V* \[*s*\] or state-action pairs *Q*\[*s, a*\]. These tabular representations store the value function di-rectly in arrays or tables indexed by states or state-action pairs. This approach is feasible when the total number of states or state-action pairs is sufficiently limited to be stored and efficiently updated in memory. Tabular methods commonly em-ploy dynamic programming techniques, such as Value Iteration and Policy Iteration, or utilize model-free learning algorithms like Monte Carlo methods and Temporal-Difference (TD) learning \[[Sutton et al., 1998](#[sutton-et-al.,-1998]-sutton,-r.-s.,-barto,-a.-g.,-et-al.-\(1998\).-reinforcement-learning:-an-introduction,-volume-1.-mit-press-cambridge.)\]. They serve as the foundational basis for popular reinforcement learning algorithms including Q-Learning \[[Watkins et al., 1989](#[watkins-et-al.,-1989]-watkins,-c.-j.-c.-h.-et-al.-\(1989\).-learning-from-delayed-rewards.-[williams,-1992]-williams,-r.-j.-\(1992\).-simple-statistical-gradient-following-algorithms-for)\] and SARSA \[[Sutton, 1995](#[sutton,-1995]-sutton,-r.-s.-\(1995\).-generalization-in-reinforcement-learning:-successful-examples-using-sparse-coarse-coding.-advances-in-neural-information-processing-systems,-8.)\], particularly when applied to relatively small-scale environ-ments.

* **Function approximation methods:** In environments characterized by large or con-tinuous state spaces (such as those involving image-based inputs or complex scenarios like chess positions) it becomes impractical to assign explicit values to each specific state. Instead, function approximation techniques are employed to estimate the value functions. These approximations are expressed mathematically as:

*V*ˆ(*s, θ*) ≈ *Vπ*(*s*)

*Q*ˆ(*s, a, θ*) ≈ *Qπ*(*s, a*)

where the parameters *θ* represent the approximator’s parameters. Typically, these approximators take as input features extracted from the state representation *ϕ*(*s*), en-abling the agent to generalize effectively to unseen or previously unvisited states. Var-ious function approximation methods exist, including linear combinations of features, neural networks, decision trees, and other advanced techniques.

2. ### **Model Based vs Model Free** {#model-based-vs-model-free}

The predictability or knowledge of the environment dynamics in a reinforcement learning problem significantly affects the choice of method:

* **Model-based:** These methods assume access to, or attempt to learn, a model of the environment. The model typically consists of the transition probabilities *P* (*s*′ | *s, a*) and the reward function *R*(*s, a*). If a model is available, the problem becomes a planning task, where the agent makes decisions by simulating future trajectories and evaluating potential outcomes without interacting with the real environment. Even when a model is not initially given, model-based approaches can learn an approximate model from experience and use it for planning (e.g., Dyna-Q, model-based policy optimization).

  * **Model-free:** In the absence of a known or learned model, the agent must rely purely on trial-and-error interactions with the environment to gather experience. These methods directly learn a policy or a value function, optimizing behavior without making explicit predictions about future states or rewards. Examples include Q-Learning, SARSA, and policy gradient methods.

3. ### **On Policy vs Off Policy**

Throughout the learning process of a reinforcement learning agent, the policy *π*(*a* | *s*) is continuously updated when attempting to improve it. Since both the state-value *Vπ*(*s*) and action-value *Qπ*(*s, a*) functions are defined *w.r.t* a given policy *π*, any change in the policy affects them. So, the manner of handling this change leads to two distinct learning paradigms:

* **On-policy learning:** Uses the data collected following the most recent policy *π* to update the value functions, thus, reflecting each policy change in real time in the learning process. One of the methods in this category is SARSA, since it generates new actions following the same policy that’s being optimized.

  * **Off-policy learning:** Maintains a separate behavioral policy *µ*, often older and more exploratory, to generate the agent’s experiences. Simultaneously, it employs a distinct target policy *π*, which is the policy being optimized and learned, as the basis for esti-mating the value functions. This decoupling allows for greater stability and efficiency in the learning process, enabling the reuse of experiences generated by previous policies, human demonstrations, or other agents. Furthermore, it facilitates extensive explo-ration without negatively impacting the refinement of the optimal policy. An example of an off-policy method is Q-Learning, which updates the action-value function based on a greedy policy while using an experience that’s generated by an *ϵ*\-greedy policy.

    4. ### **Prediction vs Control** {#prediction-vs-control}

Reinforcement learning problems are typically categorized into two fundamental types, each serving a different purpose in the learning process:

* **Prediction:** The task is to estimate the state-value function *Vπ*(*s*) or the action-value function *Qπ*(*s, a*) under a fixed policy *π*. In other words, evaluate how good a given policy is, without altering the behavior of the agent.

  * **Control:** The objective is to learn an optimal policy *π*∗ that maximizes the expected return E*π*∗ \[*G*0\]. This involves both evaluating the current policy and improving it over time through interaction with the environment.

Prediction methods serve as the foundation for control algorithms, where accurate value estimation is critical for informed policy improvement. Two key classes of model-free value estimation methods are Monte Carlo (MC) and Temporal-Difference (TD) learning, both of which rely on sampled experience to update value functions. Their main distinction lies in how returns are computed: while MC methods use the actual returns observed at the end of episodes, TD methods bootstrap by combining observed rewards with existing value estimates.

* #### **Monte Carlo (MC) methods:**

  * Estimate value functions by averaging returns observed in complete episodes.

  2. *REINFORCEMENT LEARNING PARADIGMS*	23

* Require episodes to terminate in order to compute total return *Gt* for each time-step *t*.

* Do not bootstrap. Value updates depend solely on actual observed returns.

* A typical MC update for the state-value function at each time-step *t* is of the form:

  *V* (*s*) ← *V* (*s*) \+ *α* (*Gt* − *V* (*s*))

  where *α* is the learning rate, and *s* is the state observed at time-step *t* (i.e. *St* \= *s*).

* #### **Temporal Difference (TD) learning:**

  * Estimate value functions using observed immediate rewards and bootstrap future rewards using current value estimates.

  * Can learn from incomplete episodes and apply to continuing tasks.

  * A common TD method is TD(0), with the update rule:

  *V* (*s*) ← *V* (*s*) \+ *α* (*Rt*\+1 \+ *γV* (*St*\+1) − *V* (*s*))

    where *α* is the learning rate, *γ* is the discount factor, and *s* is the state observed at time-step *t* (i.e. *St* \= *s*).

  * TD methods are the basis for control algorithms such as Q-Learning and SARSA.

Monte Carlo methods tend to have high variance but unbiased estimates, while TD meth-ods offer lower variance but introduce bias due to bootstrapping. To balance these trade-offs, the TD(*λ*) algorithm combines both approaches using eligibility traces, interpolating between MC and TD(0) depending on the parameter *λ* ∈ \[0*,* 1\].

5. ### **Value Based vs Policy Based vs Actor Critic**

The objective of a reinforcement learning agent is to learn a decision-making policy *π*∗(*a* | *s*), that would maximize the expected return E*π*∗ \[*G*0\]. To achieve this, the agent leverages the collected experience to learn in one of the following ways:

* #### **Value-based methods:**

  * Focus on estimating the action-value function *Qπ*(*s, a*)

    * The new policy is then derived by choosing, for each state, the action that maxi-mizes the value:

  *π*′(*s*) \= argmax *Q*(*s, a*)

  *a*∈*A*

      * **Policy-based methods:** Directly optimize the policy itself using the returns observed from experience, without explicitly estimating value functions.

      * **Actor-critic methods:**  Combine both approaches:

* The critic learns to evaluate the current policy.

* While the actor improves the policy using the critic’s evaluation.

  3. ## **Deep RL Algorithms** {#deep-rl-algorithms}

[Deep Reinforcement Learning (Deep RL)](#deep-rl-deep-reinforcement-learning.) algorithms fall under the category of function approximation methods, as they primarily rely on deep neural networks to represent value functions or policies. This makes them more practical and well-suited for high dimensional or complex environments. Some of the most widely known methods include:

1. ### **Deep Q-Learning (DQN)**

Deep Q-Learning (or Deep Q-Network) \[[Mnih et al., 2013](#[mnih-et-al.,-2013]-mnih,-v.,-kavukcuoglu,-k.,-silver,-d.,-graves,-a.,-antonoglou,-i.,-wier-stra,-d.,-and-riedmiller,-m.-\(2013\).-playing-atari-with-deep-reinforcement-learning.-arxiv-preprint-arxiv:1312.5602.)\] is the function approximation equivalent of tabular Q-Learning. Both are value-based methods that learn an action-value function estimation *Q*(*s, a*) from rewards received by experience to derive a greedy policy that selects actions maximizing the estimated value.

The estimation is done by updating the *Q* function towards the target:

*y* \=	*Rt*\+1	if *st*\+1 is a terminal state  
*Rt*\+1 \+ *γ* max*a Q*(*st*\+1*, a*)	otherwise

*Q*\[*st, at*\] ← *Q*\[*st, at*\] \+ *α* (*yt* − *Q*\[*st, at*\])

where *α* is the learning rate. Whereas in DQN, the *Q* function is approximated by a neural network *Q*ˆ(*s, a, θ*), parameterized by weights *θ*. The problem here becomes a regression task, where the loss is defined as:  
 *y* − *Q*ˆ(*s , a , θ*) 2

which is then minimized by updating the network parameters *θ* using gradient descent.

4. ### **AlphaZero**

AlphaZero \[[Silver et al., 2018](#[silver-et-al.,-2018]-silver,-d.,-hubert,-t.,-schrittwieser,-j.,-antonoglou,-i.,-lai,-m.,-guez,-a.,-lanctot,-m.,-sifre,-l.,-kumaran,-d.,-graepel,-t.,-et-al.-\(2018\).-a-general-reinforce-ment-learning-algorithm-that-masters-chess,-shogi,-and-go-through-self-play.-science,-362\(6419\):1140–1144.)\] is a reinforcement learning algorithm developed for the games of Chess, Shogi, and Go. It is a more generalized version of AlphaGo Zero \[[Silver et al., 2017](#[silver-et-al.,-2017]-silver,-d.,-schrittwieser,-j.,-simonyan,-k.,-antonoglou,-i.,-huang,-a.,-guez,-a.,-hubert,-t.,-baker,-l.,-lai,-m.,-bolton,-a.,-et-al.-\(2017\).-mastering-the-game-of-go-without-human-knowledge.-nature,-550\(7676\):354–359.)\], and it managed to use the same architecture and algorithm for all three of the aforementioned games without requiring any game-specific adaptations. AlphaGo Zero itself is an improved algorithm over the original AlphaGo \[[Silver et al., 2016](#[silver-et-al.,-2016]-silver,-d.,-huang,-a.,-maddison,-c.-j.,-guez,-a.,-sifre,-l.,-van-den-driessche,-g.,-schrittwieser,-j.,-antonoglou,-i.,-panneershelvam,-v.,-lanctot,-m.,-et-al.-\(2016\).-mastering-the-game-of-go-with-deep-neural-networks-and-tree-search.-nature,-529\(7587\):484–489.)\] the main difference is that AlphaGo  
Zero (and subsequently AlphaZero) uses *zero* human knowledge during training, yet it still outperforms its predecessors.

AlphaZero belongs to the category of actor-critic methods, as it trains a deep neural net-work (*π, v*) \= *fθ* that outputs, for each state *s*, a probability distribution over the actions *πθ*(· | *s*) and a value estimation *vπθ* (*s*).

At a high level, AlphaZero alternates between two main phases, executing them for a fixed number of iterations. It begins with multiple **self-play** games, which serve as the experience collection process, guided by the MCTS algorithm. This process generates training data of the form ⟨*s*; {*π, v*}⟩, which is then used in the **neural network training** phase to optimize the parameters of *fθ*.

#### **Self-Play**

Self-play consists of multiple games played by AlphaZero against itself.  The objective is to generate a dataset ⟨*st*; {*πt, z*}⟩ at various time-steps *t* across different games, where *πt*(· | *st*) is the improved action probability distribution obtained via MCTS, and *z* is the final game outcome. Since there are no intermediate rewards, *z* is the same for all states within a single game.

* *W* (*s, a*)**:** Accumulated value across simulations.

* *Q*(*s, a*)**:** Average value, computed as *W* (*s,a*) if *N* (*s, a*) *\>* 0, otherwise 0\.

5. ### **MuZero**

MuZero \[[Schrittwieser et al., 2020](#[schrittwieser-et-al.,-2020]-schrittwieser,-j.,-antonoglou,-i.,-hubert,-t.,-simonyan,-k.,-sifre,-l.,-schmitt,-s.,-guez,-a.,-lockhart,-e.,-hassabis,-d.,-graepel,-t.,-et-al.-\(2020\).-mastering-atari,-go,-chess-and-shogi-by-planning-with-a-learned-model.-nature,-588\(7839\):604–609.)\] is a model-based reinforcement learning algorithm that extends AlphaZero. Originally applied to games such as Atari, Chess, Shogi, and Go, MuZero generalizes to a broader class of environments, including those with intermediate rewards and single-agent tasks. However, it is not suited for stochastic (non-deterministic) environments. Its main contribution lies in integrating a learned model of the environment, enabling an agent that combines planning and tree search without access to a simulator.

Instead of learning to predict full environment states, MuZero learns a compact internal representation *ϕs* that facilitates accurate predictions of three core signals: value, policy, and reward. Importantly, this internal state carries no semantic alignment with the true environ-ment state *s*.

MuZero’s model comprises three key neural networks:

* **Prediction Network** (*π, v*) \= *fθ*(*ϕs*)**:** Outputs the policy and value from the internal

  3. *DEEP RL ALGORITHMS*	31

  state. Structurally similar to AlphaZero’s network, except it operates on internal states rather than observed ones.

* **Representation Network** *ϕs* \= *hθ*(*s*)**:** Encodes the environment state *s* into the initial internal state *ϕst* , used as the root node of the search tree.

* **Dynamics Network** (*Rt*\+1*, ϕst*\+1 ) \= *gθ*(*ϕst, at*)**:** Predicts the next reward and internal state given the current internal state and action.

At a high level, MuZero operates similarly to AlphaZero, with the critical difference being that MuZero learns its own model of dynamics rather than relying on a predefined simulator. This leads to key differences in the handling of:

* **State Transitions:** Each node in the search tree corresponds to an internal state *ϕs*. The dynamics model *gθ* is used to generate successor nodes *ϕst*\+1 from *ϕst* and an action *at*.

* **Legal Actions:** Because the internal state *ϕs* lacks semantic meaning, legal actions cannot be inferred during tree expansion. Thus, illegal actions are masked only at the root node (where the actual environment state *st* is known). Over time, the policy network *fθ* learns to avoid illegal actions, as they are never selected during training (action selection at the root relies on *πt*, where only legal actions are permitted).

* **Terminal States:** Internal states *ϕs* do not inherently indicate termination. Conse-quently:

  1. The value is always predicted by the network, rather than obtained from the terminal state value *v*(*sT* ) as in AlphaZero.

  2. The model may generate internal states beyond the terminal point, i.e., {*ϕsT* \+*a* | *a \>* 0}, but these are trained to produce the same predicted value *v*(*sT* ), effectively treating the terminal state *sT* as an absorbing state.

The representation network *hθ*, the dynamics network *gθ*, and the prediction network *fθ* are trained jointly in an end-to-end fashion using self-play data. During training, the envi-ronment state *st* is first encoded into a latent state *ϕst* \= *hθ*(*st*). The model then unrolls *K* hypothetical future steps by applying the dynamics network *gθ* repeatedly, producing se-quences of internal states *ϕst*\+1 *, ϕst*\+2 *, . . . , ϕst*\+*K* and corresponding reward predictions *r*ˆ*k*. At each step *k*, the prediction network *f* outputs a value *v*ˆ*k* and policy *π*ˆ*k* from the latent state *ϕst*\+*k* .

These predicted quantities (*R*ˆ*k*, *v*ˆ*k*, *π*ˆ*k*) are compared against targets obtained from MCTS and actual observed rewards during self-play. Specifically, the value targets *vk* come from the outcome of MCTS rollouts, the policy targets *πk* are the normalized visit counts from the search tree, and the reward targets *Rk* are the true rewards returned by the environ-ment. All three losses are aggregated and minimized through backpropagation across the unrolled computation graph, allowing gradients to update the parameters of *hθ*, *gθ*, and *fθ* simultaneously.

4. ## **Conclusion**

Reinforcement learning has emerged as a powerful paradigm for solving sequential decision-making problems in environments characterized by uncertainty, partial observability, and delayed feedback. From foundational tabular methods to sophisticated deep learning ap-proaches, the field has witnessed significant progress in both theoretical frameworks and practical applications. In particular, deep reinforcement learning techniques such as DQN, PPO, and AlphaZero have demonstrated remarkable success in complex domains by leverag-ing function approximation and scalable architectures.

The distinction between model-free and model-based approaches reflects different philoso-phies in how agents learn and plan. While model-free methods dominate in large-scale ap-plications due to their simplicity and generality, model-based approaches like MuZero offer promising avenues by integrating planning into the learning process without requiring explicit knowledge of the environment.

Recent advances continue to push the boundaries of what reinforcement learning can achieve, yet fundamental challenges remain, such as improving sample efficiency, generaliza-tion across tasks, and safe exploration in real-world scenarios. As the field evolves, a unified understanding of prediction, control, and planning will play a central role in developing intelligent agents that learn robustly and adaptively from interaction alone.

1. ### **Halide**

Halide \[[Ragan-Kelley et al., 2013](#[ragan-kelley-et-al.,-2013]-ragan-kelley,-j.,-barnes,-c.,-adams,-a.,-paris,-s.,-durand,-f.,-and-amarasinghe,-s.-\(2013\).-halide:-a-language-and-compiler-for-optimizing-parallelism,-locality,-and-recomputation-in-image-processing-pipelines.-acm-sigplan-notices,-48\(6\):519–530.)\] is a [Domain-Specific Language (DSL)](#dsl-domain-specific-language.) embedded in C++ that targets high-performance image and array processing on modern heterogeneous architectures, including [CPU](#cpu-central-processing-unit.)s and [GPU](#gpu-graphics-processing-unit.-hpc-high-performance-computer.-if-interleaving-factor.)s. It is designed to support both portability and efficiency by enabling automatic exploitation of memory locality, vectorization (via [SIMD](#simd-single-instruction-multiple-data.) instructions), and parallelism. Halide can generate code for various backends, such as x86, ARM, CUDA, OpenCL, and OpenGL, across platforms including macOS, Linux, Android, and Windows.

Its key innovation lies in decoupling algorithm specification from execution scheduling. The algorithm, expressed as a multistage pipeline, defines what computation should be per-formed, each stage consuming data from the previous stage and feeding into the next. In contrast, the execution schedule, written as method calls on Halide functions, specifies how the computation should be executed, including loop nesting, parallelism, and vectorization.

This separation enables developers to optimize performance without altering the algo-rithmic logic, thus facilitating experimentation with various execution strategies tailored to specific hardware. However, despite the flexibility of Halide’s scheduling language, identi-fying an efficient schedule remains a non-trivial task, often requiring expert knowledge and substantial manual tuning.

#### **Optimizing Halide with Reinforcement Learning (Halide RL)**

Halide RL \[[Pecenin et al., 2019](#[pecenin-et-al.,-2019]-pecenin,-m.,-maidl,-a.-m.,-and-weingaertner,-d.-\(2019\).-optimization-of-halide-image-processing-schedules-with-reinforcement-learning.-in-simp´osio-em-sistemas-computacionais-de-alto-desempenho-\(sscad\),-pages-37–48.-sbc.)\] is an automated optimization framework that leverages [Reinforcement Learning (RL)](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) to generate efficient execution schedules for image processing pipelines written in Halide. This hybrid approach allows domain knowledge to guide the search, while the [RL](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) agent performs fine-grained policy learning.

    * selectdep: selects and adds a strong dependency,

      * nextdep: progresses to the next available dependency.

  Each state corresponds to a partially constructed schedule. Validity is ensured through dependency constraints derived using *Farkas’ Lemma*, which encodes affine constraints into sets of linear inequalities.

      * **Exploration of the Coefficient Space (Second [MDP](#mdp-markov-decision-process.))** After constructing the schedule structure, the second [MDP](#mdp-markov-decision-process.) (see figure [3.5](#figure-3.5:-schedule-space-exploration-mdp.)) explores whether it lies within a defined *polytope* by selecting numerical coefficient values. NeuroVectorizer \[[Haj-Ali et al., 2020](#[haj-ali-et-al.,-2020]-haj-ali,-a.,-ahmed,-n.-k.,-willke,-t.,-shao,-y.-s.,-asanovic,-k.,-and-stoica,-i.-\(2020\).-neurovectorizer:-end-to-end-vectorization-with-deep-reinforcement-learn-ing.-in-proceedings-of-the-18th-acm/ieee-international-symposium-on-code-generation-and-optimization,-pages-242–255.)\] is a novel framework that addresses the challenge of automatic loop vectorization in modern compilers targeting [SIMD](#simd-single-instruction-multiple-data.) architectures. Tradi-tional compiler cost models often rely on static heuristics that fail to capture intricate data dependencies and code structure. The objective of NeuroVectorizer is to automatically de-termine the optimal [Vectorization Factor (VF)](#vf-vectorization-factor.) and [Interleaving Factor (IF)](#gpu-graphics-processing-unit.-hpc-high-performance-computer.-if-interleaving-factor.) for loops using an end-to-end deep [Reinforcement Learning](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) approach, thereby optimizing execution perfor-mance while maintaining correctness.

The [Vectorization Factor (VF)](#vf-vectorization-factor.) and the [Interleaving Factor (IF)](#gpu-graphics-processing-unit.-hpc-high-performance-computer.-if-interleaving-factor.) are two parameters that govern the vectorization of loops. The [VF](#vf-vectorization-factor.) specifies the number of operations from consecutive loop iterations that are executed in parallel as a single [SIMD](#simd-single-instruction-multiple-data.) instruction. In contrast, the [IF](#gpu-graphics-processing-unit.-hpc-high-performance-computer.-if-interleaving-factor.) determines the spacing or stride between memory accesses of successive packed instructions, enabling the vectorization of memory accesses that are not necessarily contiguous. This flexibility is particularly useful when dealing with complex memory access patterns, often referred to as non-unit stride accesses.

This transformation ensures that structurally and semantically similar loops are mapped to nearby points, enabling the reinforcement learning agent to generalize vectorization strategies across similar code patterns. The embedding process involves analyzing the abstract syntax tree of the code and extracting semantic paths that capture the logic and structure of the loop. These paths are then encoded and aggregated into a continuous vector that reflects the underlying functionality of the loop, such as memory access patterns or computational behavior.

    * **RL Agent:** The agent consumes the embedding and outputs VF and IF decisions. These decisions are automatically injected as compiler pragmas.

6. ### **MLIR RL**

MLIR RL \[[Bendib et al., 2024](#[bendib-et-al.,-2024]-bendib,-n.,-aouadj,-i.-n.,-and-baghdadi,-r.-\(2024\).-a-reinforcement-learning-environment-for-automatic-code-optimization-in-the-mlir-compiler.-arxiv-preprint-arxiv:2409.11068.)\] aims to automate the process of optimizing high-level code represented in the [MLIR](#mlir-multi-layer-intermediate-representation.) compiler framework. Manual optimization of operations like matrix multiplications, convolutions, and other linear algebra kernels is complex and error-prone, particularly due to the vast search space of transformation sequences. The proposed solution introduces a [Reinforcement Learning (RL)](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) environment tailored to the [MLIR](#mlir-multi-layer-intermediate-representation.) compiler, fo-cusing on operators used in deep neural networks (such as matrix multiplication, convolution, [ReLU](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.)..etc), aiming to train agents that learn efficient optimization strategies automatically.

#### **System Architecture**

The architecture of the proposed system centers around an [RL](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) agent trained to apply a sequence of compiler transformations. The main components include:

* **Feature Extractor:** Converts [MLIR](#mlir-multi-layer-intermediate-representation.) Linalg operations into structured numerical rep-resentations by extracting loop bounds, access matrices, operation counts, and trans-formation history.

  * **Hierarchical RL Agent:** Predicts both the transformation to apply and its associated parameters, using a policy network with a shared backbone and specialized heads.

    * **RL Environment:** Encodes the state (operation features), defines the action space (transformations), applies selected transformations, compiles the result, and measures performance to generate rewards.

#### **States**

The environment state is a structured feature vector extracted from [MLIR](#mlir-multi-layer-intermediate-representation.) Linalg opera-tions. It includes:

   * Loop metadata (bounds and steps),

     * Load/store access matrices,

     * Counts of mathematical operations (e.g., add, multiply, exp),

     * History of applied transformations.

These features are standardized and padded to fixed sizes to enable consistent input format-ting for the neural networks.

#### **Actions**

The action space consists of transformations applicable to [MLIR](#mlir-multi-layer-intermediate-representation.) operations, which are: tiling, parallelization, interchange, vectorization, img2col, and an action for no transforma-tion. Actions are represented hierarchically: first the transformation is selected, followed by its parameters (if any). This Cartesian decomposition enables efficient exploration of large action spaces.

#### **Rewards**

The agent is rewarded based on the speedup achieved over the unoptimized baseline.

*r* \= log	 *T*baseline   
*T*optimized

Two modes are supported:

* **Immediate reward:** Computes speedup after each transformation.

  * **Final reward:** Computes speedup only after the complete sequence. The latter was found to be more efficient for training.

#### **Agent and Policy Network**

The [RL](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) agent uses a [Proximal Policy Optimization (PPO)](#ppo-proximal-policy-optimization.) algorithm and consists of:

* **Backbone network:** A [Multi-Layer Perceptron (MLP)](#mlp-multi-layer-perceptron.) processing input features into a latent feature vector, which is then dispatched to the transformation and parameters networks.

  * **Transformation network:** Predicts which transformation to apply.

    * **Parameters networks:** Specialized sub-networks for predicting: tile sizes for both parallelization and tiling, and permutations for interchange, depending on the selected transformation.

    * **Value network:** Estimates the expected return from a given state.

This hierarchical policy architecture enables the agent to efficiently reason about both discrete and parameterized actions.

2. ## **MLIR Code**

[MLIR](#mlir-multi-layer-intermediate-representation.) features a rich set of dialects, each having a different level of abstraction and a specific set of operations for a particular domain of application. Using [MLIR](#mlir-multi-layer-intermediate-representation.) passes, we can transform a code using any set of dialects into another that uses a different set of lower-level dialects such as the LLVM dialect, which can be directly transformed into machine code for execution.

To target [Loop Nest Optimization](#lno-loop-nest-optimization.), the system requires dialects containing a set of opera-tions that can fully represent and support loop nests and their transformations. Additionally, these representations need to be high-level enough so that the system can extract meaningful and useful information to eventually help the [Deep RL](#deep-rl-deep-reinforcement-learning.) agent reason about its optimization decisions. With these constraints, three different dialects are pertinent: [SCF](#scf-structured-control-flow.), Affine, and [Linalg](#linalg-linear-algebra.). We chose working with [Linalg](#linalg-linear-algebra.) since it provides transformations for all the [Loop Nest](#lno-loop-nest-optimization.) [Optimization](#lno-loop-nest-optimization.) our system requires, and is natively built to represent affine loop nests, which

54	*CHAPTER 4\.  DESIGN AND IMPLEMENTATION*

is the exact aim of this project.

Therefore, our system operates exclusively on loops that are written as [Linalg](#linalg-linear-algebra.) operations. When passed to the system, the code is scanned for all existing [Linalg](#linalg-linear-algebra.) operations, each is parsed to extract its features (as will be explained in Section [4.4](#in-summary,-the-environment-facilitates-the-interaction-between-the-rl-agent-and-the-mlir-compiler-by-managing-benchmarks,-applying-transformations,-handling-errors-robustly,-evaluating-performance,-and-enabling-sequential-optimization-across-multiple-operations-within-complex-programs.)), and then transforma-tions are applied to them in sequence. Each applied transformation implies a change to the operation at hand in the code.

3. ## **Environment**

The [Reinforcement Learning](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) environment serves as the crucial interface between the learn-ing agent and the [MLIR](#mlir-multi-layer-intermediate-representation.) compiler infrastructure. It is responsible for managing the state of the optimization process, executing the agent’s chosen actions ([MLIR](#mlir-multi-layer-intermediate-representation.) transformations), eval-uating the performance impact, and providing feedback in the form of rewards. The design encapsulates the complexities of interacting with the compiler, presenting a standardized RL interface (*reset*, *step*) to the agent.

1. ### **Initialization and Benchmark Management**

Upon initialization, the environment creates unique temporary files to avoid conflicts, loads each benchmark’s [MLIR](#mlir-multi-layer-intermediate-representation.) code and its baseline execution time, extracts the relevant [Linalg](#linalg-linear-algebra.) operation features for state representation (Section [4.4](#in-summary,-the-environment-facilitates-the-interaction-between-the-rl-agent-and-the-mlir-compiler-by-managing-benchmarks,-applying-transformations,-handling-errors-robustly,-evaluating-performance,-and-enabling-sequential-optimization-across-multiple-operations-within-complex-programs.)), and applies a preprocessing step that converts any *conv2d* into an equivalent matrix–matrix multiplication. This setup prepares a pool of characterized codes for the agent to learn from.

2. ### **Interaction Loop**

The environment implements the standard RL interaction cycle, which consists of two main functions: *reset* which starts new episodes, and *step* which receives an action from the agent and returns a new state as well as a feedback. We detail the design of these two functions in the following sections:

It selects a benchmark, either randomly (during experience collection) or by a specific index (during evaluation), from the loaded benchmarks. Crucially, it sets the initial state to correspond to a specific operation within that benchmark, since a benchmark can contain multiple operations. Operations are processed in reverse order, typically beginning with the last operation.

  * **Step:** Executes a single step based on the agent’s selected action. Given the current state and a raw action from the agent, it performs the following sequence:

    1. **Action Processing:** The action is interpreted into a specific [MLIR](#mlir-multi-layer-intermediate-representation.) transfor-mation and its parameters. This maps the agent’s abstract choice to a concrete compiler operation (detailed in Section [4.5](#this-state-representation-helps-the-agent-make-an-informed-decision-based-on-the-poly-hedral1-features,-memory-accesses,-and-computation-complexity-of-the-current-operation.)).

  3. *ENVIRONMENT*	55

     2. **Transformation Application:** Attempts to apply the processed transformation to the current [MLIR](#mlir-multi-layer-intermediate-representation.) code, and returns the resulting code after transformation.

        3. **Transformation Error Handling:** If the transformation is invalid or fails within the [MLIR](#mlir-multi-layer-intermediate-representation.) infrastructure, the step terminates early.            4. **State Update:** The environment updates the state to reflect all the changes after applying the transformation, including the operation features, the action mask, and the transformed code.

5. ## **Actions Space**

In order for the [RL](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) agent to change the environment and attempt to optimize the code, an action space needs to be defined, which allows to specify what actions could the agent select, and how to select them. In our case, the actions correspond to a set of [LNO](#lno-loop-nest-optimization.) transformations that are supported by [MLIR](#mlir-multi-layer-intermediate-representation.) through its *transform* dialect. The set of supported transfor-mations O is: **tiling**, **parallelization**, **interchange**, **vectorization**, and a **no transfor-mation** action that allows the agent to stop the optimization sequence before the step limit is reached.

Most of these transformations require additional parameters for their definition to be complete (for example, tile sizes for the tiling transformation). As a result, in our case, actions are actually a combination of the transformation *o* and its parameters *x*, for this we introduce, instead of the action probability *π*(*a* | *s*), the following notation:

*π*(*o, x* | *s*) \= *Pπ*(*Ot* \= *o, Xt,o* \= *x* | *St* \= *s*)

where *π* represents the policy, *St* is the state observed at time-step *t*, *Ot* denotes the [Loop](#lno-loop-nest-optimization.) [Nest Optimization](#lno-loop-nest-optimization.) transformation selected at time-step *t*, and *Xt,o* refers to the parameters sequence selected at time-step *t* for transformation *o* (which mean that the event *Xt,o* \= *x* can still be decomposed into sub-events). Following this notation, the probability of selecting a transformation *o* is written as:

*π*(*o,* · | *s*) \=

*x*∈*χ*(*o*)

*π*(*o, x* | *s*)

where *χ*(*o*) is the set of all possible parameters sequences applicable to transformation *o*.

1. ### **Hierarchical Action Space**

If we were to use a single probability distribution over every possible transformation–parameters combination ⟨*o, x*⟩, the size of the action space would become drastically large, making it dif-ficult to learn an effective distribution.

To address this, we factor the joint probability of selecting a transformation and its parameters sequence into two independent probabilities. Which means that:

*π*(*o, x* | *s*) \= *Pπ*(*Ot* \= *o, Xt* \= *x* | *St* \= *s*)

\= *Pπ*(*Ot* \= *o* | *St* \= *s*) · *Pπ*(*Xt* \= *x* | *St* \= *s, Ot* \= *o*)  
(4.1)  
\= *Pπ*(*Ot* \= *o* | *St* \= *s*) ·	*Pπ*(*Xt,i* \= *xi* | *St* \= *s, Ot* \= *o*)

*i*∈|*x*|

By factoring the probabilities in this way, we can define two hierarchical levels in the action space:

1. **Transformation Selection:** Given state *s*, choose which transformation *o* to apply, according to *Pπ*(*Ot* \= *o* | *St* \= *s*).

2. **Parameters Selection:** Given state *s* and the chosen transformation *o*, in order to form the parameters sequence *x*, we select its elements *xi* according to *Pπ*(*Xt,i* \= *xi* | *St* \= *s, Ot* \= *o*), where *i* is the position of the element in the sequence.

#### **Tile Sizes**

When applying loop tiling to a chosen loop nest, the agent must select a positive tile size for each loop level. To facilitate this, the agent independently chooses a tile size from a predefined candidate set at each loop level. This candidate set (whose size is controlled by a hyperparameter) contains positive divisors of the loop’s upper bound (beginning with 1), plus the value 0 to represent the option of *no tiling* (as illustrated in Figure [4.4](#figure-4.4:-parallel/non-parallel-tile-sizes-selection.)).

#### **Parallel Tile Sizes**

In order to perform parallelization in [MLIR](#mlir-multi-layer-intermediate-representation.), a tiling transformation should be applied first, and then, the resulting outer loops are marked for parallelization. Therefore, the agent must choose explicit tile sizes to mark for parallelization. The parameterization of parallel tile sizes follows the same procedure as standard tiling: at each loop level, the agent selects a tile size from the candidate set of divisors of the loop’s upper bound plus 0 to indicate *no parallelization*. With the difference being that for reduction loop levels the candidates are omitted and only the 0 one is kept, since reduction loops shouldn’t be parallelized. Moreover, despite their resemblance, the parameterization for tiling and for parallelization are independent from one another, since they correspond to two distinct transformations.

This ensures that, even though the state representation remains unchanged during intermediate steps, the agent can distinguish which sub-action to produce next.

6. ## **Reward Signal**

In reinforcement learning, the agent’s policy is shaped by the reward signal it receives from the environment. Because our objective is to minimize execution time through code op-timizations, the reward must accurately reflect improvements (or regressions) in performance. We therefore decompose the reward into two components:

1. ### **Execution Speedup**

This component quantifies the end-to-end performance gain and is only computed once, after all transformations have been applied and the optimized code has been successfully executed. We compare the execution time of the optimized code against the original baseline:

*R*	\= log  initial exec time	 *.*

By deferring this reward until the end of the episode, we ensure:

* The final code version (incorporating all transformations) is evaluated as a whole.

  * Temporary speedups followed by regressions do not produce misleading intermediate rewards.

    * The system does not need to compile and execute code after each transformation, significantly reducing training time and computational overhead.

    2. ### **Transformation Errors** {#transformation-errors}

Since each transformation can introduce compilation or execution failures, we penalize any step that results in an invalid code state. Specifically:

* If a transformation fails to apply correctly, the agent receives a reward of −5 immedi-ately.

  * If, at the end of the episode, the optimized code either fails to execute or produces incorrect results, the agent receives a reward of −20\.

This error-penalization strategy discourages the agent from exploring transformation se-quences that compromise correctness or prevent successful execution.

In all other intermediate steps (where no error or final speedup applies), the agent receives a reward of 0\.

In this section, we provide an end-to-end description of the design of the [Deep RL](#deep-rl-deep-reinforcement-learning.) agent, covering both its neural network architecture and the learning algorithm used to train it.

1. ### **Network Architecture**

Our agent employs an actor–critic design realized by two separate, yet structurally similar, feedforward networks. Both the policy (“actor”) and value (“critic”) networks share the same input processing and embedding pipeline before diverging into specialized output heads.

#### **Input Layer**

The input is a fixed-length feature vector (see Section [4.4](#in-summary,-the-environment-facilitates-the-interaction-between-the-rl-agent-and-the-mlir-compiler-by-managing-benchmarks,-applying-transformations,-handling-errors-robustly,-evaluating-performance,-and-enabling-sequential-optimization-across-multiple-operations-within-complex-programs.)) passed to botht the actor and the critic. As a preprocessing step, the loop bounds in the input are normalized in order to reduce their magnitude, since they tend to take large values compared to the other input values.

#### **Backbone**

A two-layer fully connected module (512 neurons each with ReLU activations), followed by a 512-neuron output, transforms the preprocessed inputs into a compact latent represen-tation. This embedding helps capture intricate nonlinear interactions among features and is similar but separate in both the actor and critic.

#### **Critic Head**

A single linear neuron takes the embedding and outputs the state-value estimate *V* (*s*).

#### **Actor Heads**

The policy network branches into four heads, each producing a probability distribution for one component of the hierarchical action:

1. **Transformation Head**

A fully connected layer of size |O| (the number of supported transformations), followed by softmax, to yield *π*(*o,* · | *s*).

2. **Tiling Head**

Structured as (max loops) independent softmax groups (one per loop level), each over (max tiles) candidates (including a *no-tiling* option), to select tile sizes for standard tiling.

3. **Parallel-Tiling Head**

Mirrors the Tiling Head’s structure but governs parallel-tiling choices independently, using the same group-wise softmax configuration.

4. **Interchange Head**

   Defined by the chosen interchange method:

   * **Enumeration:** A softmax layer over (3 · max loops − 6\) discrete swap candidates.

   * **Level Pointers:** A softmax over (max loops) pointer positions, with already-selected loops masked out during sub-actions.

   * **Continuous Encoding:**  Two outputs:  a mean *µ* from a linear layer and a global learnable standard deviation *σ*, which parameterize a Gaussian policy over \[0*, n*\!).

This design leverages a shared representation for efficiency while providing specialized heads that accommodate the discrete, continuous, and hierarchical nature of our transforma-tion actions.

#### **Proximal Policy Optimization**

[PPO](#ppo-proximal-policy-optimization.) was chosen for the following reasons:

* **Clipped Surrogate Objective:** By enforcing a soft trust-region on policy updates, the clipping mechanism prevents overly large steps, which is important especially when advantage estimates introduce high variance due to sparse rewards. This clipping en-sures gradual and reliable improvements.

  * **Mixed Action-Space Support:** [PPO](#ppo-proximal-policy-optimization.) natively handles both discrete decisions (trans-formation and parameters selection) and continuous outputs (for the *continuous encod-ing* method; see Section [4.5.2](#figure-4.4:-parallel/non-parallel-tile-sizes-selection.)), matching actions design.

    * **Sample Efficiency via GAE:** Generalized Advantage Estimation \[[Schulman et al., 2015b](#[schulman-et-al.,-2015b]-schulman,-j.,-moritz,-p.,-levine,-s.,-jordan,-m.,-and-abbeel,-p.-\(2015b\).-high-dimensional-continuous-control-using-generalized-advantage-estimation.-arxiv-preprint-arxiv:1506.02438.)\] combines multi-step [TD](#td-temporal-difference.) errors with Monte Carlo returns through exponentially decay-

  ing weights. Tuning the *λ* parameter allows us to balance bias and variance, making each trajectory more informative.

      * **Reduced Hyperparameter Sensitivity:** [PPO](#ppo-proximal-policy-optimization.) demonstrates robust performance across a wide range of learning rates, clipping thresholds, and batch sizes, minimizing the effort required for tuning.

      * **Scalability and Parallel Rollouts:** Its design supports concurrent environment in-stances, which is critical when each step involves code compilation and execution. Par-allel rollouts significantly shorten training time.

      * **Proven Empirical Performance:** [PPO](#ppo-proximal-policy-optimization.) consistently achieves strong results in diverse domains and benefits from mature, well-maintained implementations in standard [RL](#relu-rectified-linear-unit.-rl-reinforcement-learning.-rpc-remote-procedure-call.) libraries.

  To integrate [PPO](#ppo-proximal-policy-optimization.) into our system, we made two key adaptations:

      * **Trajectory Collection:** [PPO](#ppo-proximal-policy-optimization.) typically assumes a fixed trajectory length. We there-fore enforce an upper limit on the number of transformations per episode and collect a constant number of benchmark evaluations per trajectory. However, because the agent can terminate early by selecting the *no transformation* action, the actual number of steps may vary across trajectories. The reason the number of transformations is con-strained by an upper bound rather than a fixed value, is that rewards are provided only at the end of each episode, consequently, it is essential that all training trajectories are complete.

8. ## **Levels Pointers vs Continuous Encoding vs Enu-merated Candidates for interchange**

In this section, we evaluate the impact of our proposed interchange methods (Levels Point-ers and Continuous Encoding) against the traditional Enumerated Candidates approach. As discussed in Section [1.3.3](#parallelization-for-a-specific-loop-makes-use-of-hardware-capabilities-to-execute-its-itera-tions-simultaneously-by-splitting-them-between-multiple-threads-or-processes-for-multi-core-and-multi-processor-architectures.-for-this-transformation-to-be-legal-iterations-being-paral-lelized-have-to-be-independent,-i.e.-those-iterations-don’t-have-any-data-dependency-between-them.-figure-1.2-demonstrates-the-impact-of-using-parallelization-on-a-loop.), loop interchange is a critical transformation in performance tun-ing, especially in loop-heavy domains such as neural networks and scientific simulations. It affects data locality, cache usage, and vectorization potential.

The standard method enumerates a portion of valid permutations of loop orders and al-lows the agent to pick one. While this is conceptually simple, it misses out on a large number of possible permutations, which limits some potential optimizations. Our Levels Pointers and Continuous Encoding approaches are able to cover all the space of possible permutations, by leveraging technics like sequence-generation and continuous actions.

11. ## **Conclusion**

In this chapter, we have thoroughly evaluated our proposed [Deep RL](#deep-rl-deep-reinforcement-learning.)\-based optimization system across two distinct computational domains: neural network operators and [LQCD](#lqcd-lattice-quantum-chromodynamics.) benchmarks. The empirical results highlight the strengths, generalizability, and practical trade-offs of our system, demonstrating significant performance improvements compared to baseline and competitive auto-schedulers.

Through systematic experimentation, we showed that our [PPO](#ppo-proximal-policy-optimization.) agent consistently achieves notable speedups, particularly on compute-intensive workloads, where it efficiently leverages transformations such as loop interchange, tiling, parallelization, and vectorization. The detailed ablation studies further revealed critical insights about the optimal interchange method, with the Levels Pointers approach providing the best scalability and performance trade-off.

Additionally, comparing [PPO](#ppo-proximal-policy-optimization.) against AlphaZero highlighted the complementary strengths of both algorithms: while AlphaZero excelled on smaller or more irregular kernels due to its deep exploration capabilities, [PPO](#ppo-proximal-policy-optimization.) was superior in optimizing structured, computationally intensive loops due to its efficient policy learning.

The comparative analysis with state-of-the-art auto-schedulers established the robustness and competitiveness of our [PPO](#ppo-proximal-policy-optimization.)\-based solution, achieving execution times comparable to or better than specialized and exhaustive search approaches without the associated overhead.

In summary, our results validate the efficacy of using [Deep Reinforcement Learning](#deep-rl-deep-reinforcement-learning.), par-ticularly [PPO](#ppo-proximal-policy-optimization.) combined with Levels Pointers interchange, as a unified and practical frame-work for automated compiler optimization. Future work will focus on enhancing the agent’s context awareness, exploring hybrid strategies that blend PPO and AlphaZero, and extend-ing the optimization pipeline to handle broader classes of scientific and machine learning applications.

# **Conclusion** {#conclusion-1}

In this work, we have presented a [Deep Reinforcement Learning](#deep-rl-deep-reinforcement-learning.) framework for automatic code optimization in the [MLIR](#mlir-multi-layer-intermediate-representation.) compiler that is domain-independent. Our contributions span both breadth and depth: we extended the scope of [Deep RL](#deep-rl-deep-reinforcement-learning.)\-based optimization beyond conventional deep learning kernels by integrating the demanding [LQCD](#lqcd-lattice-quantum-chromodynamics.) domain, demon-strating that a single trained agent can generalize across radically different computational patterns. To make this possible, we redesigned the action space (most notably by introduc-ing the Levels Pointers and Continuous Encoding interchange mechanisms) which allows the agent to navigate the factorial complexity of loop reordering without exploding the action set. We also explored a new algorithmic frontier by adapting both [Proximal Policy Opti-mization (PPO)](#ppo-proximal-policy-optimization.) \[[Schulman et al., 2017](#[schulman-et-al.,-2017]-schulman,-j.,-wolski,-f.,-dhariwal,-p.,-radford,-a.,-and-klimov,-o.-\(2017\).-proximal-policy-optimization-algorithms.-arxiv-preprint-arxiv:1707.06347.)\] and AlphaZero \[[Silver et al., 2018](#[silver-et-al.,-2018]-silver,-d.,-hubert,-t.,-schrittwieser,-j.,-antonoglou,-i.,-lai,-m.,-guez,-a.,-lanctot,-m.,-sifre,-l.,-kumaran,-d.,-graepel,-t.,-et-al.-\(2018\).-a-general-reinforce-ment-learning-algorithm-that-masters-chess,-shogi,-and-go-through-self-play.-science,-362\(6419\):1140–1144.)\] to this problem, comparing their strengths and trade-offs in real compiler workflows. Finally, we rigorously benchmarked our system against state-of-the-art auto-schedulers that leverage heuristics, machine learning, and exhaustive search, showing that our learned policies deliver compet-itive or superior end-to-end speedups on both neural network operators and full [LQCD](#lqcd-lattice-quantum-chromodynamics.) codes.

Despite these advances, several limitations remain. First, our current state representation is operation-centric and does not capture inter-operation interactions or global program struc-ture, as a result, we treat multi-operation codes as sequences of independent optimizations, which can miss cross-operation dependencies. Second, the sparse, end-of-episode reward (while efficient) hides the contribution of individual transformations and can slow conver-gence on very deep or irregular nests. Third, the overhead of repeated compilation and execution still constrains training scale, making it challenging for the [Deep RL](#deep-rl-deep-reinforcement-learning.)\-agent to col-lect larger experiences or train for longer epochs in order to learn better.

Looking forward, these challenges suggest several promising directions. Enriching the state with coarse-grain program-level features or inter-operation dependencies could enable truly holistic optimization strategies. Hybridizing [PPO](#ppo-proximal-policy-optimization.)’s fast training time and generaliz-ability with AlphaZero’s deeper lookahead and stable convergence may combine their best qualities. Leveraging learned cost-models or predictive performance estimators could re-duce the need for full recompilation, dramatically accelerating training. Finally, extending the framework to offer additional [Loop Nest Optimization](#lno-loop-nest-optimization.)s (such as loop fusion and [GPU](#gpu-graphics-processing-unit.-hpc-high-performance-computer.-if-interleaving-factor.) parallelization) will give it the potential for better unmatched performances.

# **Appendix** {#appendix}

## **General Architecture**

The [LQCD](#lqcd-lattice-quantum-chromodynamics.) compiler, developed by [MIT](#mit-massachusetts-institute-of-technology.), is a python-based compiler that translates high level [LQCD](#lqcd-lattice-quantum-chromodynamics.) code into an architecture-specific executable format. It consists of two main phases:

* **Frontend:** Performs lexical, syntactic, and semantic analysis of the [LQCD](#lqcd-lattice-quantum-chromodynamics.) code, pro-ducing an internal [LQCD](#lqcd-lattice-quantum-chromodynamics.) [IR](#ir-intermediate-representation.) (See Section [5](#figure-5.18:-lqcd-compiler-\(without-lno\).-the-frontend-parses-and-analyzes-high-level-lqcd-code-into-an-intermediate-lqcd-ir.-the-backend-originally-translates-this-ir-to-halide-ir.-we-have-extended-it-with-a-halide-to-mlir-translator-\(the-resulting-mlir-code-can-be-fed-directly-into-our-deep-rl-based-system\)-and-then-lowered-to-llvm-to-be-compiled-and-executed-on-target-hardware.)).

* **Backend:** Lowers the [LQCD](#lqcd-lattice-quantum-chromodynamics.) [IR](#ir-intermediate-representation.) to a target-specific representation and emits exe-cutable code, applying domain-aware optimizations during this process.

Our interest is on the backend part of this compiler since it is the part responsible for how the code is lowered eventually. It was originally designed to translate the [LQCD](#lqcd-lattice-quantum-chromodynamics.) [IR](#ir-intermediate-representation.) to a Halide [IR](#ir-intermediate-representation.) (See Section [5](#figure-5.19:-lqcd-ir-syntax.)) that is executed afterwards using Halide runtime. The [MIT](#mit-massachusetts-institute-of-technology.) team wanted to explore [MLIR](#mlir-multi-layer-intermediate-representation.) as an alternative to Halide, to benefit from its robust design, rich set of transformations and optimizations, and wide range of supported hardware architectures.

In order to lower to [MLIR](#mlir-multi-layer-intermediate-representation.) code we further extended this backend to translate Halide [IR](#ir-intermediate-representation.) to [MLIR](#mlir-multi-layer-intermediate-representation.) specifically into a set of [Linalg](#linalg-linear-algebra.) generic operations, excluding the need for the Halide backend. The resulting [MLIR](#mlir-multi-layer-intermediate-representation.) code is then simply passed to our system to be optimized and executed. This transformation is particularly effective, as most Halide pipelines can be expressed using only [Linalg](#linalg-linear-algebra.) generic operations.

## **LQCD IR**

The [LQCD](#lqcd-lattice-quantum-chromodynamics.) [IR](#ir-intermediate-representation.) serves as an [Intermediate Representation](#ir-intermediate-representation.), offering a unique language tai-lored to express the complex mathematical computations and algorithms fundamental to simulating particle behavior on a lattice. It breaks down complex programs into simpler operations. This [IR](#ir-intermediate-representation.) has a syntax that can express all the computations used in [LQCD](#lqcd-lattice-quantum-chromodynamics.), so this section won’t dive into the details of these structures and operations since it is not what this project’s about.

## **Halide IR**

The Halide [IR](#ir-intermediate-representation.) serves as an [Intermediate Representation](#ir-intermediate-representation.), between the [LQCD](#lqcd-lattice-quantum-chromodynamics.) [IR](#ir-intermediate-representation.) and the Halide environment. It’s the result of the [LQCD](#lqcd-lattice-quantum-chromodynamics.) [IR](#ir-intermediate-representation.)’s transformation by the backend. It allows the transformation code to be compiled and executed in the Halide environment.

Figure [5.20](#figure-5.20:-halide-ir-syntax.) presents every possible expression defined in the Halide IR syntax, the com-bination of these expressions creates a Halide pipeline. The Halide pipeline is the structure that describes how the code should execute in Halide. The following points present each element constructing the [IR](#ir-intermediate-representation.):

* **sym**: the symbolic name for a variable in the IR, it contains the “name” specified by the user of the LQCD language, and a “stamp”, a sequential number. The combination of the name and stamp creates a unique identifier for the variable.

* **var**: represents a Halide Var, which is a simple variable that can used in Halide Func (functions). This type contains one attribute which is “name” of type sym.

* **rdom**: represents a range domain in Halide, it’s basically multiple ranges (intervals) defined each one by two bounds. So this type has two attributes: “name” for the symbolic name of the range domain and “bounds” as a list of range types. Each range has a lower and upper bound.

* **type**: represents the standard data type for variables. This type is a union between 3 types: “F” for float, “I” for integer and “UI” for unsigned integer. Each one with the attribute “bits” which is an integer to specify the number of bits for the type (E.g., F(64) is a float with 64 bits).

* **param**: abbreviation for parameter, represents an input buffer for the Halide pipeline, it’s just an input for our program, it’s passed to the compiler by the user. For its attributes, it has a “name”, the symbolic name, “ndim” an integer for defining the dimension of the input and “type” for its type. * **out**: abbreviation for output, represents an output buffer for the Halide pipeline, it’s the result of our program, it’s returned by the compiler to the user. For its attributes, it has a “name”, the symbolic name, “ndim” an integer for defining the dimension of the input and “type” for its type.

* **func**: abbreviation for function, represents a Halide Func, which is a function that can be executed in Halide, it has a “name”, symbolic name, and “ndim” an integer that defines the dimension of function’s output.

* **op**: abbreviation for operation, represents an arithmetic operation. This type is a union between “Mult” for multiplication, “Add” for addition, “Div” for division and “Sub” for subtraction.

## **MLIR  Generation**

The goal of our contribution is to lower the Halide [IR](#ir-intermediate-representation.) input to [MLIR](#mlir-multi-layer-intermediate-representation.) [Linalg](#linalg-linear-algebra.) generic operations such that the transformed code would have the same loop structure as if it was lowered using Halide’s backend. Therefore, the design of our contribution was dictated by the way computations are performed within Halide. It consists of different consecutive stages:

1. **Convert Function Definitions:** bodies of Halide functions definitions (Halide Pure-Defs and Halide Updates) are converted into [MLIR](#mlir-multi-layer-intermediate-representation.) code. The resulting codes are then registered to be used later when generating loop nests bodies.

2. **Build Loop Nests Structure:** defines the hierarchy and the order of loop nests as well as their sizes. Each function definition is a loop nest unless it is inlined (which happens when the function is not a producer[1](#1producers-and-consumers-are-defined-following-several-rules-in-halide-including:-compute-instructions-\(compute-at-and-compute-root-\)-and-update-definitions-scheduling.-a-producer-is-the-loop-nest-that-fills-a-buffer-with-values-that-are-used-by-another-loop-nest-\(the-consumer\).)). To find sizes of loop nests (which is commonly referred to as *bounds inference*), the backend needs to know, for each function definition, the values that could be taken by its arguments. This is achieved by analyzing the expressions used as arguments, in order to find the intervals of their values which are then used as loop nests sizes. The order and hierarchy are defined by the producer-consumer[1](#1producers-and-consumers-are-defined-following-several-rules-in-halide-including:-compute-instructions-\(compute-at-and-compute-root-\)-and-update-definitions-scheduling.-a-producer-is-the-loop-nest-that-fills-a-buffer-with-values-that-are-used-by-another-loop-nest-\(the-consumer\).) relationships.

3. **Generate [Linalg](#linalg-linear-algebra.) Operations:** generates the final code for each loop nest as a [Linalg](#linalg-linear-algebra.) generic operation following the structure built in the previous stage, along with alloca-tion code for intermediate buffers needed to store values computed by loop nests.

1Producers and consumers are defined following several rules in Halide including: compute instructions (*compute at* and *compute root* ) and update definitions scheduling. A producer is the loop nest that fills a buffer with values that are used by another loop nest (the consumer).
